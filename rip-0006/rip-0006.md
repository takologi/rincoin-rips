## Abstract

This RIP specifies the **Cryptographic Vault (CV)**: a keyless autonomous smart contract that quarantines UTXOs whose underlying signature scheme has crossed the cryptographic event horizon (τ_c ≈ 30 years), shielding them from adversarial extraction (Type B loss) while preserving their identity for legitimate owner recovery via zero-knowledge proofs.

The CV bifurcates the thermodynamic Event Horizon defined in RIP-0005 into two sub-phases: a Vulnerable Latent Phase (assets dormant less than τ_c, still under user key control) and a Shielded Latent Phase (assets dormant beyond τ_c, held within the CV awaiting either ZKP-authenticated recovery by their original owner or, ultimately, PoR combustion at τ).

The CV is keyless by design: no master key, no admin role, no upgrade authority. Custody is governed entirely by deterministic consensus rules and ZKP verification.

## Motivation

### The cryptographic obsolescence problem

The Proof of Rinne mechanism in RIP-0005 enforces a thermodynamic statute of limitations of τ ≈ 400 years. This horizon greatly exceeds the expected secure lifespan of contemporary elliptic-curve cryptography:

- secp256k1 (the curve used for transparent and MWEB Pedersen commitments) is widely considered to be quantum-vulnerable on a timescale of 30–100 years given current trends in fault-tolerant quantum computer development.
- Even classical cryptanalytic improvements may erode security margins on a 100-year horizon.

If a UTXO sits dormant past the cryptographic event horizon, an adversary capable of breaking secp256k1 could extract the assets *before* they reach PoR maturity. Without intervention, the protocol's regenerative guarantees collapse into an extraction race.

### Why a single master key is unacceptable

A naïve solution—designate a multisig committee to "rescue" vulnerable UTXOs into a quantum-secure address—fails because:

- The multisig itself becomes a systemic honeypot, attracting both adversarial and political pressure.
- Custodianship contradicts Rincoin's trustless design.
- Quantum attacks on the rescue keys would simply shift the failure point.

The CV resolves this by being **keyless**: no party (including the Rincoin Core Authority) holds the ability to extract assets from the Vault outside of the consensus-defined autonomous expulsion path.

### Why ZKP recovery is essential

A purely mechanical CV (no recovery mechanism) would penalize legitimate long-term holders whose deep-cold-storage keys are still valid but whose addresses crossed τ_c due to inactivity. ZKP recovery allows such holders to prove ownership without exposing their (possibly compromised in the quantum sense) original private keys to the network.

## Specification

### 1. Parameters

| Parameter | Symbol | Value | Notes |
|---|---|---|---|
| Cryptographic event horizon | τ_c | 15,768,000 blocks (~30 years) | Initial; governable per §6 |
| Cryptographic Vault Time threshold | τ_V | τ_blocks - τ_c | Maturation period within Vault before PoR-combustion |
| ZKP recovery fraction (target) | γ_mean | 10% | Long-run macro target; not enforced per-event |
| Recovery cooldown | — | 30 days | Per UTXO; prevents rapid replay |

### 2. The bifurcated Event Horizon

A UTXO's lifecycle progresses through:

```
[ Active Use ]                          (last_spend < current)
      ↓ (no spend for > 0 blocks)
[ Vulnerable Latent Phase ]             (dormant 0 < t < τ_c, user key control)
      ↓ (dormancy crosses τ_c)
[ Cryptographic Event Horizon ]         (forced extraction trigger)
      ↓
[ Shielded Latent Phase / Vault ]       (custody by CV smart contract)
      ↓ (CVT reaches τ_V)
[ Combustion ]                          (PoR Reincarnation per RIP-0005)
```

A UTXO leaves the Vulnerable Latent Phase via one of two paths:

- **Voluntary migration**: The user spends the UTXO to a quantum-secure address before reaching τ_c. The UTXO never enters the Vault.
- **Forced extraction**: At dormancy = τ_c, if the UTXO has not been voluntarily migrated, it is forcibly extracted and routed into the CV via the Sweeper Bounty mechanism (RIP-0007).

### 3. The Cryptographic Vault smart contract

The CV is a single, well-known contract address derived deterministically from the genesis-defined CV identifier:

```
CV_ADDRESS = HASH256("rincoin.cryptographic.vault.v1") truncated and bech32m-encoded
```

The exact derivation is provided in the Reference Implementation. The CV address:

- Has no spending key. Outputs sent to it cannot be spent except via the consensus-defined autonomous expulsion path or via successful ZKP recovery.
- Is whitelisted at the consensus layer as a destination for forced extraction outputs.
- Maintains internal state (a Merkle tree of original-owner public-key commitments) that is part of consensus state.

### 4. Vault state structure

Upon ingestion of a forcibly extracted UTXO, the CV records:

```
struct VaultEntry {
    uint256  original_pubkey_hash;       // hash of the original transparent address pubkey
    uint64   value_satoshis;             // asset value
    uint64   ingestion_block_height;     // CVT t=0
    uint64   sweeper_bounty_paid;        // bounty disbursed to sweeper, per RIP-0007
    bytes32  merkle_branch_commit;       // commitment to inclusion in vault state tree
}
```

Vault entries are organized into a Merkle Mountain Range (MMR) structure for efficient append and inclusion-proof generation. The MMR root is committed in every Layer-1 block header (as a new commitment field, soft-fork-compatible with non-CV-aware nodes).

### 5. Cryptographic Vault Time (CVT)

CVT for a Vault entry is measured in Layer-1 block heights:

```
CVT(entry) = current_block_height - entry.ingestion_block_height
```

The autonomous expulsion condition is:

```
CVT(entry) >= τ_V
```

When this condition is satisfied, the entry is expelled from the CV and its value is added to the Rinnechain reserve `R_in` per RIP-0005. The expulsion is automatic and does not require a transaction.

The CVT mechanism replaces any reliance on external time oracles. All time is measured in confirmed block heights.

### 6. Zero-Knowledge Proof Recovery

A legitimate owner of a UTXO that has been forcibly extracted into the CV MAY recover a fraction of the asset value by submitting a valid ZKP. The procedure:

#### 6.1 Off-chain proof generation

The owner generates a zk-SNARK or zk-STARK proof attesting:

```
PROOF: ∃ (sk, sig) such that
    PubKey(sk) = entry.original_pubkey_hash      AND
    Verify(sk, sig, recovery_message) = true
```

where `recovery_message` is a hash binding the recovery to:

- The Vault entry (to prevent replay against other entries),
- A newly-generated quantum-secure recovery address `R_addr`,
- A nonce drawn from current chain state (to prevent precomputation).

The original private key `sk` is **never** broadcast.

#### 6.2 On-chain submission

The owner submits a transaction containing:

- The ZKP `π`.
- The Merkle inclusion proof for `entry` in the current CV MMR.
- The recovery address `R_addr` (must be on a quantum-secure curve when such curves are activated by future RIPs; until then, MAY be a standard transparent address).

The transaction is processed at the consensus layer:

```
verify_recovery(π, entry_proof, R_addr, recovery_message):
    require entry exists in CV MMR (Merkle proof valid)
    require entry not previously recovered (no double-claim)
    require recovery_message correctly bound to entry, R_addr, chain nonce
    require π is valid for the public statement
    if all conditions hold:
        compute recovery_amount = floor(entry.value_satoshis * γ_recovery_rate)
        transfer recovery_amount from CV → R_addr
        mark entry.recovered = true (consensus state update)
```

#### 6.3 Recovery rate

The per-event recovery rate `γ_recovery_rate` is fixed at the consensus layer:

```
γ_recovery_rate = 0.10    (10% of entry value)
```

The remaining 90% stays in the CV and ultimately combusts at CVT = τ_V.

The rationale for partial (rather than full) recovery is discussed in §Rationale.

#### 6.4 Recovery deadline

ZKP recovery MUST occur during the Vault custody window:

```
0 < CVT(entry) < τ_V
```

After CVT = τ_V, the entry is expelled to PoR combustion and recovery is no longer possible.

### 7. ZKP scheme requirements

The ZKP scheme used for recovery MUST satisfy:

- **Soundness**: A computationally bounded prover without `sk` cannot generate a valid proof except with negligible probability.
- **Zero-knowledge**: The proof reveals no information about `sk` beyond the validity of the statement.
- **Succinctness**: Proof size SHOULD be O(log n) or smaller relative to the witness size, to keep transaction sizes manageable.
- **Trusted setup**: If using zk-SNARKs, the trusted setup ceremony MUST be a multi-party computation with at least 100 participants, and the parameters MUST be published prior to consensus activation. Implementations using zk-STARKs avoid this requirement.

The specific ZKP scheme is NOT mandated by this RIP; conforming implementations may use any scheme satisfying the above. The reference implementation will use **Halo2** or **Plonky2** (both zk-SNARK constructions with universal trusted setup) based on benchmarking results to be documented in the Reference Implementation.

### 8. Activation

This RIP activates at the same hard fork as RIP-0005 (PoR), namely at `t_max ≈ 444 years post-genesis`. However, for assets to enter the Vault, the cryptographic event horizon τ_c (~30 years) must have been reached for at least some UTXOs—which occurs much earlier, at approximately year 30 post-genesis.

To accommodate this timing mismatch, the **Vulnerable Latent Phase tracking and Sweeper Bounty mechanics (RIP-0007)** activate independently at year 30 post-genesis (block height ≈ 15,768,000), via a separate hard fork specified in RIP-0007. The full Vault custody and ZKP recovery infrastructure activates with this RIP at year 444.

In the intervening period (years 30 to 444), the protocol operates as follows:

- Forcibly extracted UTXOs are aggregated into a "pre-Vault" address that behaves identically to the CV but does not yet support ZKP recovery (recovery infrastructure may not yet be cryptographically ready).
- All such pre-Vault entries are migrated automatically into the Vault at PoR activation, with their CVT preserved.

This staged activation is necessary because mature post-quantum ZKP infrastructure may not exist at the year-30 boundary. Future RIPs MAY accelerate ZKP recovery activation if the cryptographic landscape develops faster than anticipated.

### 9. τ_c governance

τ_c is governable, similar to τ in RIP-0005:

- `τ_c_min = 7,884,000 blocks` (15 years)
- `τ_c_max = 26,280,000 blocks` (50 years)

Adjustments use the same PID controller framework as RIP-0005's τ governance, with the target objective being "maintain τ_c sufficiently below the projected post-quantum threat horizon while not prematurely quarantining still-secure assets."

A τ_c adjustment requires Rinnechain epoch consensus and is bounded:

- Single-adjustment magnitude: `|Δτ_c| ≤ 525,600 blocks` (1 year).

## Rationale

### Why 10% recovery rate?

The 10% target derives from the macro-equilibrium analysis in [1, §6.5]. Under the assumption that ~10% of dormant assets represent legitimate long-horizon holders (rather than truly lost keys), the post-recovery equilibrium becomes:

```
C^ss_ZKP = r_f / [μ(1-β)(1-γ_mean)]
```

At μ = 1.7%, β = 2%, γ_mean = 10%, this yields C^ss_ZKP ≈ 21.03M RIN, matching the canonical Nakamoto target.

The 10% is a per-event rate applied uniformly (not a quota). Some events may have 0% recovery (no owner is alive/aware); others may have higher legitimate ownership. The 10% rate is set such that the *expected* aggregate recovery aligns with the macro target.

### Why partial recovery (10%) rather than full?

Three reasons:

1. **Reduces incentive for fraudulent recovery**: An attacker who has somehow obtained the original private key (e.g., via quantum break) has only 10% to gain rather than 100%. The other 90% combusts to the network's benefit.
2. **Compensates the network for security cost**: The network bore the cost of indefinitely protecting the dormant asset within the CV. The 90% retention compensates for this service.
3. **Ensures macro-economic equilibrium**: The 10%/90% split is calibrated to produce the target steady state.

### Why keyless rather than multisig?

A multisig CV would require:

- A persistent committee to hold keys.
- A governance procedure to rotate the committee over centuries.
- Trust that the committee will not collude.

None of these is sustainable on a multi-century horizon. The keyless design has zero key holders by construction, eliminating these concerns.

### Why MMR rather than a flat list?

The CV is expected to ingest millions of entries over its lifetime. Verifying inclusion of a single entry must be efficient (O(log n)) for ZKP recovery to be practical. A Merkle Mountain Range provides efficient append (O(log n)) and inclusion proof (O(log n)) with append-only semantics. Modifications (marking an entry as recovered) are handled via a separate "spent" bit-vector commitment in consensus state.

### Alternatives considered

1. **Burn vulnerable UTXOs immediately at τ_c (no Vault)** — Rejected. Penalizes legitimate owners with no recovery path. Violates property-rights expectations.
2. **Government-style forfeiture (require active "claim" within τ_c)** — Rejected. Imposes unrealistic burden on long-horizon holders. The whole point of cold storage is *not* to require active maintenance.
3. **Re-issue assets to original address with new (post-quantum) signatures** — Rejected. The protocol cannot verify the original owner controls the address without a recovery mechanism, which is exactly what ZKP recovery provides.
4. **Auction-based recovery (highest bidder wins)** — Rejected. Captures the asset for capital, not for the rightful owner.

## Backwards Compatibility

This RIP defines a **hard fork** with staged activation:

- Year 30: Sweeper Bounty mechanism (RIP-0007) activates. Forcibly extracted UTXOs flow to a pre-Vault address.
- Year 444 (PoR activation): Full CV consensus rules, ZKP recovery, and CVT-based expulsion activate. Pre-Vault entries migrate to the Vault.

Implementations not carrying CV logic post-activation will:

- Fail to validate forced-extraction transactions (will reject them as non-standard sends to a "burn" address with consensus implications).
- Fail to honor ZKP recovery transactions.
- Diverge from the CV-aware chain.

- TV-1: A UTXO at dormancy = τ_c is forcibly extracted with sweeper bounty paid per RIP-0007, residual to CV.
- TV-2: A valid ZKP recovery for a Vault entry transfers exactly 10% to the recovery address.
- TV-3: A double-claim ZKP recovery is rejected (entry marked recovered).
- TV-4: A CVT >= τ_V entry is expelled to PoR R_in automatically at the next block.
- TV-5: An invalid ZKP (manipulated proof bytes) is rejected.
- TV-6: A recovery to an expired entry (CVT > τ_V) is rejected.

## Security Considerations

### Quantum threat model

The CV's primary purpose is to defend against an adversary who can break secp256k1. The following scenarios are within scope:

- **Pre-τ_c quantum break**: An adversary breaks secp256k1 before τ_c. Vulnerable UTXOs may be extracted by the adversary before forced extraction triggers. Mitigation: voluntary migration to quantum-secure addresses (Phase A of the migration in RIP-0008) is strongly incentivized during years 20–30. τ_c may be adjusted downward via governance if the threat materializes early.
- **Post-τ_c quantum break (in-Vault)**: An adversary breaks secp256k1 while assets are in the Vault. The original public keys are committed in the Vault's MMR, so the adversary could in principle generate valid ZKPs against them. Mitigation: ZKP recovery is rate-limited to 10%; combined with the recovery cooldown and the requirement that the recovery address be quantum-secure (when post-quantum addresses are activated by future RIPs), the attack value is bounded.

### ZKP soundness as a security boundary

The CV's safety against fraudulent recovery rests entirely on the soundness of the ZKP scheme. A soundness break would allow an adversary without `sk` to generate accepting proofs and extract 10% of any Vault entry. Mitigations:

- Use of well-studied schemes (Halo2, Plonky2).
- Independent cryptographic audit before activation.
- Bounded loss (10% per entry, not 100%) limits the damage of a soundness break.
- Recovery cooldown (30 days per entry) limits the rate at which fraudulent recoveries can drain the Vault.

### Trusted setup ceremony (if applicable)

If the chosen ZKP scheme requires a trusted setup, the ceremony itself becomes a critical component. The setup parameters MUST be:

- Generated via multi-party computation with ≥100 participants drawn from diverse jurisdictions and economic interests.
- Published with full participant attestations.
- Subject to public review for at least 6 months before consensus activation.

Implementations using transparent ZKP schemes (e.g., zk-STARKs) avoid this requirement entirely and are RECOMMENDED over schemes requiring trusted setup, all else equal.

### Side-channel leakage from recovery patterns

If only ~10% of recovery attempts succeed (because most dormant addresses are truly lost), but legitimate-owner recovery attempts succeed nearly 100% of the time, then the *pattern* of successful recoveries leaks information about which addresses had legitimate owners.

This is not directly exploitable in most threat models, but it does mean that the privacy of legitimate long-horizon holders is reduced upon recovery. Holders concerned about this leakage SHOULD use the L2 Account Abstraction layer (specified by future RIPs) for additional privacy.

### Centralization pressure on ZKP infrastructure

ZKP proof generation may require specialized hardware (GPUs, FPGAs). If the recovery process becomes computationally infeasible on commodity hardware, only well-resourced holders can recover. This is a centralization concern.

Mitigations:

- The recovery is not time-critical (a holder has years between τ_c and τ to generate the proof).
- Recovery-as-a-service (delegated proof generation) is permissible, provided the delegate cannot extract `sk`. ZKP schemes that allow proof outsourcing (e.g., via additional zero-knowledge wrapping) are RECOMMENDED.
