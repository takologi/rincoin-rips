## Abstract

This RIP specifies the implementation strategy for the Customized Halving (CH) subsidy schedule defined in RIP-0002, replacing network-specific hardcoded block-height constants in `GetBlockSubsidy` with a single height formula expressed as multiples of `nSubsidyHalvingInterval`. The result is one phase-resolution code path shared by mainnet, testnet, and regtest, eliminating an entire class of per-network divergence bugs and making `GetBlockSubsidy` a verifiably pure function of block height.

---

## Motivation

RIP-0002 defines the CH schedule abstractly in terms of phase boundaries. A direct implementation hardcodes the boundary heights for each network — for example, mainnet boundaries at 840000 / 2100000 / 4200000 / 6300000 and separate regtest values. This duplication is a recurring source of defects:

- A correct mainnet schedule can silently disagree with regtest, so simulation does not faithfully validate production behaviour.
- Adding or relocating a phase requires editing every network branch.
- Boundary-value tests must be re-derived per network.

Because each CH boundary is, by construction, an integer multiple of `nSubsidyHalvingInterval`, the entire schedule can be expressed once as a function of that single per-network constant. The same expression then governs all networks; only `nSubsidyHalvingInterval` differs.

---

## Specification

### 1. Subsidy Schedule (from RIP-0002)

Let `I = nSubsidyHalvingInterval` (mainnet 210,000; regtest 210).

| Phase | Height range | Subsidy (RIN) | Boundary |
|-------|--------------|---------------|----------|
| 0–3 | `0 .. 4I − 1` | standard Bitcoin-style halving (50 → 6.25) | — |
| 4 (CH) | `4I .. 10I − 1` | 4.00 | `h >= 4I` |
| 5 | `10I .. 20I − 1` | 2.00 | `h >= 10I` |
| 6 | `20I .. 30I − 1` | 1.00 | `h >= 20I` |
| Terminal | `h >= 30I` | 0.60 | `h >= 30I` |

On mainnet (`I = 210,000`): CH dilation at 840,000; Phase 5 at 2,100,000; Phase 6 at 4,200,000; Terminal at 6,300,000.

---

### 2. Implementation

`GetBlockSubsidy` (`validation.cpp`) evaluates the schedule using `nSubsidyHalvingInterval` multipliers, highest boundary first:

```cpp
const int interval = consensusParams.nSubsidyHalvingInterval;

if (nHeight >= 30 * interval) return  0.60 * COIN; // Terminal
if (nHeight >= 20 * interval) return  1.00 * COIN; // Phase 6
if (nHeight >= 10 * interval) return  2.00 * COIN; // Phase 5
if (nHeight >=  4 * interval) return  4.00 * COIN; // Phase 4 (CH)

// nHeight < 4 * interval: standard pre-CH halving schedule
```

(Subsidy values shown in RIN for clarity; the implementation uses satoshi: 0.60 RIN = 60,000,000 sat, etc.)

The pre-CH branch (`nHeight < 4 * interval`) retains the standard geometric halving (50 → 25 → 12.5 → 6.25 RIN) unchanged.

---

### 3. Determinism Guarantee

`GetBlockSubsidy` MUST depend only on `nHeight` and `nSubsidyHalvingInterval`. It MUST NOT consult chain state, caches, or any value that varies with reorg history. As a consequence, after a reorg of arbitrary depth the subsidy at any height is recomputed identically — the function is pure.

---

## Rationale

Expressing boundaries as `k * nSubsidyHalvingInterval` rather than literal heights yields three properties:

1. **Single code path.** Mainnet and regtest exercise identical logic; regtest simulation is a faithful proxy for mainnet behaviour.
2. **Reorg safety by construction.** Purity over `nHeight` means deep reorgs cannot corrupt the subsidy schedule; correctness does not depend on sequential application of blocks.
3. **Testability.** Boundary-value analysis is network-independent: the same `k` multipliers locate every boundary on every network.

---

## Backwards Compatibility

This RIP changes only the *implementation* of the schedule already specified by RIP-0002; it does not alter any mainnet subsidy value at any height. The mainnet emission curve is identical before and after this change. It is bundled into the Block 840,000 hard fork release for delivery convenience and is co-activated with RIP-0009.

## Verification

Reorg resilience and boundary correctness are demonstrated by:

- `scripts/sim-ch-rin3.sh` — BVA at all eight boundary blocks (839/840, 2099/2100, 4199/4200, 6299/6300) plus four escalating reorg attacks (1, 1261, 2101, 2101 blocks).
- `scripts/sim-ch-attack.sh` — Omega attack: a 5,461-block reorg invalidating Block 840 from height 6,300, erasing Phases 4–6 and Terminal, then re-mining and confirming every boundary subsidy is restored exactly.

In all runs the rolled-back and restored subsidies match the expected satoshi values with zero failures, empirically confirming the purity guarantee of §3.

Proof screenshots (`sim-ch-rin3.sh` v4 PASS: 35/0, `sim-ch-attack.sh` pending execution) will be published in the [rincoin-sim repository](https://github.com/Aevust/rincoin-sim) after the feature branch is merged.

## Security Considerations

- **Pure-function invariant.** Any implementation that reads chain state, cached values, or any quantity that varies with reorg history inside `GetBlockSubsidy` violates the determinism guarantee of §3. A violation can cause a reorged node to compute a different subsidy than a node that never experienced the reorg, leading to a consensus split. The function MUST be a pure mapping from `(nHeight, nSubsidyHalvingInterval)` to `CAmount`.

- **Integer arithmetic only.** All phase-boundary comparisons are integer multiples of `nSubsidyHalvingInterval`. Floating-point arithmetic MUST NOT be used at any point in the subsidy computation; rounding errors could cause off-by-one satoshi divergence at boundary blocks.

- **Boundary ordering.** The implementation evaluates boundaries highest-first (Terminal before Phase 6 before Phase 5 before Phase 4). Reversing the evaluation order would produce incorrect subsidies for every post-CH phase. Implementations MUST preserve this ordering.
