## Abstract

This RIP specifies the **Phased Legacy Address Migration Protocol** governing the transition of UTXOs from cryptographic schemes deemed obsolete (initially: secp256k1-based ECDSA/Schnorr) into either quantum-secure replacement addresses (voluntary path) or the Cryptographic Vault (compulsory path), as the cryptographic event horizon τ_c approaches.

The protocol defines two activation phases:

- **Phase A (Voluntary Migration Window)**: A 10-year window prior to τ_c during which holders are progressively warned and incentivized to migrate.
- **Phase B (Forced Extraction)**: At τ_c, all unmigrated UTXOs become Sweep Candidates per RIP-0007.

To prevent network-load spikes from synchronized forced extraction, this RIP adopts the **Temporal Smoothing** approach using a ~144,000-block (~100-day) sliding eligibility window. Alternative approaches considered and rejected include parallel block structures and dedicated migration sidechains.

## Motivation

### The synchronization problem

If all UTXOs created in the same era reach τ_c at approximately the same time (as is the case for genesis-era UTXOs, all created in the first few months of network operation), forced extraction would produce a sweep storm:

- Network bandwidth saturates with sweep transactions.
- Mempool congestion delays normal transactions.
- The bounty market collapses (all sweepers compete for the same brief window).

A smoothing mechanism is required.

### The communication problem

A holder whose UTXO crosses τ_c without warning may experience this as confiscation. Even though the Cryptographic Vault preserves the asset for ZKP recovery (RIP-0006 §6), the user experience of "your assets disappeared from your wallet" is unacceptable.

A phased migration with progressive on-chain warning beacons gives holders the opportunity to migrate before forced extraction.

### Operational coordination

Wallet implementations need consistent rules for when to display migration warnings, when to suggest migration transactions, and when to mark addresses as "deprecated." Encoding these rules into a normative RIP ensures consistent UX across implementations.

## Specification

### 1. Phase A: Voluntary Migration Window

For each UTXO `u`, Phase A spans:

```
[creation_block(u) + τ_c - 5,256,000,  creation_block(u) + τ_c)
```

i.e., the 5,256,000-block (~10 year) period immediately preceding the cryptographic event horizon. During Phase A:

- The UTXO remains spendable under its legacy key.
- Wallet implementations MUST display a migration warning when the UTXO is observed.
- The warning urgency MUST escalate as the UTXO approaches τ_c (suggested thresholds: yellow at >5 years remaining, orange at 1–5 years, red at <1 year).
- The protocol provides a **migration transaction discount** (specified in §3) to incentivize voluntary migration.

#### 1.1 Migration transaction definition

A migration transaction:

```
TX_MIGRATE:
  vin:  [u]                                             # legacy-key UTXO
  vout: [migrated_value → new_quantum_secure_address]   # full or partial value
        [optional change → original_address or new_addr]
  signatures: standard ECDSA/Schnorr under legacy key
```

The transaction MUST direct the migrated value to an address using a quantum-secure cryptographic scheme. Until such schemes are activated by future RIPs, this requirement is non-binding (any standard transparent address is acceptable). A future RIP defining post-quantum address formats will enforce the requirement.

### 2. Phase B: Forced Extraction

At block height `creation_block(u) + τ_c`, UTXO `u` enters Phase B and becomes a Sweep Candidate per RIP-0007. Sweepers may immediately extract `u`, routing the residual to the CV.

The transition from Phase A to Phase B is instantaneous at the τ_c boundary, with one critical caveat: **temporal smoothing** modifies the *order* in which UTXOs become eligible, but not the underlying age criterion.

### 3. Migration transaction discount

To incentivize voluntary migration during Phase A, the protocol applies a fee discount to migration transactions:

```
fee_discount(blocks_until_tau_c) = max(0, min(0.50, 0.50 × (1 - blocks_until_tau_c / 5,256,000)))
```

The discount linearly increases from 0% at the start of Phase A (10 years before τ_c) to 50% at the end of Phase A (immediately before τ_c). The discount is applied as a credit to the network fee, computed at the consensus layer.

The discount MUST be claimed in the same block at the same coinbase. Specifically, the block subsidy is augmented by the total discounts claimed in that block, ensuring the discount does not deplete the macro supply schedule but redistributes it from late-Phase-A migrants to miners.

### 4. Temporal Smoothing

Without intervention, all UTXOs created in block range `[a, a+W)` would simultaneously enter Phase B at block `[a+τ_c, a+τ_c+W)`. For genesis-era UTXOs, this could place tens of thousands of UTXOs in Phase B over a few weeks.

To distribute the load, the protocol applies a **deterministic eligibility delay**:

```
sweep_eligible_height(u) = creation_block(u) + τ_c + smoothing_offset(u)

where:
    smoothing_offset(u) = HASH(u.txid || u.vout) mod 144,000
```

The offset is a deterministic pseudorandom value in the range `[0, 144,000)` Layer-1 blocks (~100 days). Each UTXO's eligibility is delayed by its individual offset, distributing the sweep load uniformly across a ~100-day window.

Sweepers MUST verify the offset before submitting a sweep transaction; a sweep before `sweep_eligible_height(u)` is rejected. The bounty mechanism (RIP-0007) is otherwise unchanged.

### 5. Beacon emissions during Phase A

To facilitate wallet detection of migration urgency, the protocol emits **Migration Beacon** events. A beacon is a special log entry produced at three Phase A milestones:

```
PHASE_A_MILESTONE_5YR:  block height = τ_c - 2,628,000
PHASE_A_MILESTONE_1YR:  block height = τ_c - 525,600
PHASE_A_MILESTONE_30D:  block height = τ_c - 43,200
```

Wallets observing these milestones for any held UTXO SHOULD escalate user notification appropriately. The beacons are advisory; consensus does not enforce wallet behavior, but full nodes MUST emit the events for wallet discovery.

### 6. Activation

Phase A semantics activate at the same block as RIP-0007: mainnet block **15,768,000** (~year 30). Specifically:

- For UTXOs created in the first 5,256,000 blocks (~years 0–10), Phase A "begins" at activation but is effectively retroactive (warnings begin immediately for these long-dormant UTXOs).
- For UTXOs created after block 5,256,000, Phase A begins at the appropriate block height per §1.

### 7. Fallback: per-block sweep cap

If, despite temporal smoothing, sweep transactions in a given block exceed a threshold of **1,000 sweep transactions per block**, additional sweeps in that block are deferred to subsequent blocks (priority queue ordered by `sweep_eligible_height`, ties broken by transaction hash). This is an emergency safety valve and is not expected to trigger under nominal load with the temporal smoothing in place.

## Rationale

### Why a 10-year Phase A?

10 years is empirically calibrated to:

- Be long enough that estate planning and inheritance procedures (which operate on multi-year cycles) have a realistic window to act.
- Be short enough relative to τ_c (~30 years) that the warning is meaningfully timed to the cryptographic event horizon, not perpetual.
- Match common institutional planning horizons for cryptographic asset audits.

A shorter window (e.g., 1 year) would surprise long-dormant holders. A longer window (e.g., 25 years) would produce premature alarm and dilute the urgency signal.

### Why temporal smoothing rather than a parallel block structure?

The Whitepaper [1, §6.4.4] discusses two approaches to the synchronization problem:

1. **Parallel block structure** — A dedicated extension block (similar to MWEB's structure per RIP-0004) reserved exclusively for migration and sweep transactions. Migration traffic does not compete with normal transactions for base-layer block space.
2. **Temporal smoothing** — A deterministic eligibility delay that spreads sweep load across a ~100-day window without changing block structure.

This RIP adopts **temporal smoothing** because:

- It preserves block-format simplicity. No new extension block type is required.
- It interoperates cleanly with RIP-0004 (MWEB), avoiding extension-block conflicts.
- It is implementable as a small consensus rule addition rather than a substantial block-format change.
- The whitepaper's analysis [1, §6.4.4] favors temporal smoothing as the lower-complexity option.

The parallel block structure approach is REJECTED by this RIP. Future RIPs MAY revisit if temporal smoothing proves inadequate.

### Why a deterministic offset rather than VRF or randomized?

A deterministic offset (computed from txid + vout) is:

- Verifiable by all nodes without coordination.
- Predictable for wallet UX (a holder can compute their UTXO's exact eligibility height).
- Resistant to manipulation by miners (the offset is fixed at UTXO creation time).

A VRF or block-randomized offset would introduce additional complexity without meaningful benefit.

### Why 50% maximum fee discount?

A 50% maximum discount is sufficient to materially incentivize migration while not creating extreme fee market distortions. A 100% discount (free migration) would risk:

- Spam migrations (rapidly migrating and re-migrating to manipulate balances or addresses).
- Significant miner revenue loss in the final months of Phase A.

50% is empirically a reasonable balance. The parameter MAY be governed in future RIPs.

### Why beacons rather than wallet polling?

Beacons are emitted as on-chain events that wallets can subscribe to via standard log/event infrastructure. The alternative—wallet polling for UTXO age—works but increases wallet implementation complexity and bandwidth. Standardized beacons promote consistent UX across implementations.

### Alternatives considered

1. **No Phase A (sweep at τ_c with no warning)** — Rejected. Confiscatory user experience.
2. **Indefinite Phase A (warnings forever, no forced extraction)** — Rejected. Defeats the purpose of cryptographic protection.
3. **Variable τ_c per UTXO based on cryptographic scheme** — Considered. Future RIPs MAY introduce per-scheme τ_c when post-quantum schemes coexist with secp256k1 on-chain. For this RIP, a single τ_c applies.
4. **Migration mandate (consensus-level prohibition of post-Phase-A spends)** — Rejected. Phase B handles unmigrated UTXOs via the sweeper mechanism (RIP-0007); a hard prohibition adds no security value.

## Backwards Compatibility

This RIP defines a **hard fork** activating at block 15,768,000 (jointly with RIP-0007).

- Implementations not carrying this RIP's logic will:
  - Fail to apply the migration fee discount, treating all transactions as standard.
  - Fail to enforce the temporal smoothing delay, accepting sweep transactions at the simple `creation + τ_c` threshold.
  - Fail to emit migration beacons.

These divergences are consensus-relevant: a non-compliant node would accept blocks that a compliant node rejects (e.g., a sweep transaction submitted before its smoothed eligibility height) and vice versa (e.g., a migration transaction with the discount applied).

## Reference Implementation

A complete reference implementation is **NOT YET AVAILABLE**. Required components:

1. Migration transaction recognition and fee-discount application.
2. Temporal smoothing offset computation and validation.
3. Beacon emission at Phase A milestones.
4. Per-block sweep cap enforcement.
5. Wallet integration patterns (recommended, not consensus-mandated).

Implementation is anticipated within the year-25 to year-30 window (5 years before activation).

## Security Considerations

### Migration window denial-of-service

If an adversary attempts to flood the network with bogus migration transactions to exhaust the discount budget, the impact is bounded: the discount is funded from the augmented coinbase, so the cost is borne by miners (who can prioritize legitimate migrations) rather than from the network budget. The 50% cap further limits the per-transaction subsidy.

### Smoothing offset predictability

Because the offset is deterministic from `(txid, vout)`, an adversary observing the UTXO set today can predict exactly when each future sweep window will open. This is a feature, not a vulnerability: it allows sweeper infrastructure to plan capacity.

The deterministic offset does NOT facilitate front-running: the sweep transaction itself still requires a valid signature (under the legacy key, or via a quantum-broken signature post-cryptographic-failure). The eligibility timing is public; the signing capability is not.

### Wallet UX failure modes

If wallet implementations fail to display Phase A warnings (or display them poorly), users may not migrate in time. Mitigations:

- Beacon emission ensures the *protocol-level* signal is unambiguous; UX failures are wallet-implementation issues, not consensus issues.
- The Cryptographic Vault (RIP-0006) provides a 30-year recovery window after Phase B begins, so even users who miss Phase A entirely have recourse via ZKP recovery.

### Migration to non-quantum-secure addresses

This RIP's migration transactions only require the destination be "an address," not specifically a quantum-secure one (because post-quantum address formats are not yet activated). A user migrating during Phase A may inadvertently migrate from one secp256k1 address to another, gaining no actual cryptographic security upgrade.

Wallet implementations SHOULD warn against this and SHOULD prefer to migrate to the most quantum-resilient address scheme available at the time. A future RIP will MANDATE post-quantum destinations once such schemes are consensus-activated.

### Interaction with MWEB

MWEB UTXOs (RIP-0004) use Pedersen commitments derived from secp256k1, and are therefore subject to the same cryptographic event horizon. However, the Phase A migration mechanism for MWEB UTXOs is non-trivial: migrating a confidential UTXO to a different cryptographic scheme requires a privacy-preserving migration protocol that does not yet exist.

This RIP defers MWEB migration handling to a future RIP. Until then, MWEB UTXOs SHOULD be voluntarily peg-out'd to transparent addresses (per RIP-0004) before approaching τ_c, then migrated under this RIP's standard procedure.

### Sweep cap as DoS surface

The per-block sweep cap (1,000 transactions) is a fallback emergency mechanism. An adversary cannot exploit it directly to delay legitimate sweeps because:

- Eligibility ordering is deterministic (priority by `sweep_eligible_height`).
- Tie-breaking by transaction hash prevents adversarial reordering.

The cap may incidentally be exceeded under genuine load spikes, in which case sweepers experience deferred processing—not loss of funds.
