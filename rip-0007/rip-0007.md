## Abstract

This RIP specifies the **Sweeper Bounty Mechanism**: a permissionless economic incentive that compensates network participants ("sweepers") for forcibly extracting UTXOs whose dormancy has crossed the cryptographic event horizon τ_c, as defined in RIP-0006. The mechanism uses a **piecewise-linear bounty function** with floor and ceiling clamps to maintain economic viability across the full range of UTXO sizes—from dust to whale.

A sweep transaction routes the bulk of the extracted UTXO value to the Cryptographic Vault (per RIP-0006), retains a small bounty fraction for the sweeper (recirculating it into active supply), and leaves no residual under the legacy private key.

## Motivation

### The dust and whale problem

A naïve flat-percentage bounty (e.g., β = 2% of UTXO value) suffers from two opposing failure modes:

1. **Dust UTXOs**: For UTXOs where 2% of value is less than the gas/fee cost of the sweep transaction, no rational sweeper will perform the sweep. These UTXOs remain at the cryptographic event horizon, exposed to adversarial extraction.
2. **Whale UTXOs**: For UTXOs of very large value (e.g., 1,000,000 RIN), 2% (= 20,000 RIN) is an outsized payout that creates extreme MEV competition and excessive value leakage from the regenerative system.

A robust mechanism must accommodate both extremes.

### Permissionless extraction is necessary

The forced extraction at τ_c cannot rely on a designated extractor (multisig, foundation, etc.), for the same reasons the Cryptographic Vault is keyless: any designated party becomes a systemic risk. Permissionless sweeping with adequate incentive distributes the workload across the network.

### MEV awareness

The Whitepaper [1, §1.1] explicitly acknowledges that the sweep events constitute structurally bounded MEV: ordering competition for sweep transactions is unavoidable, but the *harmfulness* is contained because the value extracted is determined by the bounty function (not by user-facing trade ordering). This RIP formalizes that bounded-MEV property.

## Specification

### 1. Sweep eligibility

A UTXO `u` becomes a **Sweep Candidate** at block height `h_eligible(u)`:

```
h_eligible(u) = creation_block(u) + τ_c
```

where `τ_c = 15,768,000 blocks` (~30 years; governable per RIP-0006 §9).

For all `h >= h_eligible(u)` until the UTXO is swept (or recovered legitimately), `u` is a valid sweep target.

### 2. Sweep transaction format

A sweep transaction:

```
TX_SWEEP:
  vin:  [u]                                # the sweep candidate UTXO
        [optional sweeper-funded inputs for fee coverage]
  vout: [bounty_amount → sweeper_address]  # bounty per §3
        [residual_amount → CV_ADDRESS]     # residual to Cryptographic Vault
  signatures:
        Standard ECDSA/Schnorr signature of u using the legacy private key OR
        Adversarial signature (if quantum break has occurred)
```

Critical consensus rule: a sweep transaction is valid **only if** the sender presents either (a) a valid signature under the UTXO's legacy key, or (b) any valid sequence of stack data that the consensus rules would have accepted at UTXO creation time. The protocol does not distinguish between legitimate-owner sweeps and adversarial-key-break sweeps; both result in identical Vault custody.

In other words: even if an adversary breaks secp256k1 and constructs a valid signature, they can only redirect the value to the Cryptographic Vault (modulo the bounty fraction) — not to themselves. The CV's custody is unconditional given a valid sweep transaction.

### 3. Piecewise-linear bounty function

The bounty paid to the sweeper is computed deterministically from the UTXO value:

```
bounty(v) = clamp(β · v,  bounty_floor,  bounty_ceiling)
          = max(bounty_floor, min(β · v, bounty_ceiling))
```

where:

- `v` = UTXO value in satoshis.
- `β` = bounty rate parameter, **fixed at 2% (200 basis points)** post-activation. Governable per §6.
- `bounty_floor` = `k · gas_cost`, where `k = 5` and `gas_cost` is the prevailing minimum sweep transaction fee (computed by the consensus layer at the time of sweep). The factor of 5 ensures a meaningful margin over fees.
- `bounty_ceiling` = `B_cap = 100,000 RIN` (1 × 10¹³ satoshis). The whale clamp.

The residual amount routed to the CV is:

```
residual(v) = v - bounty(v)
```

If `bounty(v) > v` (i.e., for UTXOs smaller than the floor), the bounty is reduced to `v - 1 satoshi`, with 1 satoshi to the CV (preserving the invariant that some residual always reaches the CV). UTXOs valued at 0 or 1 satoshi are exempt from sweeping.

### 4. Bounty function regimes

The piecewise structure produces three regimes:

| UTXO value range | Bounty | Effective rate | Notes |
|---|---|---|---|
| `v < bounty_floor / β` | `bounty_floor` (capped at v−1) | up to 100% | "Dust regime" — bounty floor dominates |
| `bounty_floor/β ≤ v ≤ bounty_ceiling/β` | `β · v` | exactly β (2%) | "Linear regime" — proportional to value |
| `v > bounty_ceiling/β` | `bounty_ceiling` | < β | "Whale regime" — ceiling dominates |

For the default parameters (β = 2%, B_cap = 100,000 RIN), the linear regime spans:

- Lower edge (depending on gas_cost): typically v ≥ ~5,000 satoshis × 5 / 0.02 ≈ 1.25 RIN.
- Upper edge: v ≤ 5,000,000 RIN.

UTXOs above 5,000,000 RIN (extreme whales) trigger the ceiling. There are very few such UTXOs in any plausible Rincoin distribution, so the whale clamp's macro impact is bounded.

### 5. Activation

This RIP activates at mainnet block height **15,768,000** (~30 years post-genesis, i.e., the first block at which UTXOs created at genesis reach the cryptographic event horizon).

Activation is automatic and time-deterministic. No signaling is used.

### 6. Parameter governance

The parameters β, B_cap, and k are governable via the same Rinnechain epoch governance procedure used for τ_c (RIP-0006 §9). Bounds:

- `β ∈ [0.01, 0.05]` (1% to 5%)
- `B_cap ∈ [10,000, 1,000,000] RIN`
- `k ∈ [3, 20]`

A single Rinnechain epoch may adjust β by at most 0.005 (50 basis points).

The PID controller's target objective for these parameters is the macro-equilibrium ratio defined in RIP-0005 §6.1 (R_in : Pending : C_Rin ≈ 77 : 70 : 21).

### 7. Pre-RIP-0006 operation (years 30 to 444)

Per RIP-0006 §8, full Cryptographic Vault infrastructure activates at PoR activation (year ~444). In the intervening period, the residual flows to a **pre-Vault** address that:

- Behaves as a black-hole consensus address (sends accepted, no spends possible).
- Is migrated en bloc to the CV at PoR activation, with each pre-Vault entry's `ingestion_block_height` preserved.

The sweeper bounty mechanics in this RIP are identical regardless of whether the residual flows to the pre-Vault or the full CV.

## Rationale

### Why piecewise-linear rather than purely percentage?

A pure percentage bounty fails at both ends of the value spectrum, as discussed in Motivation. The piecewise structure with floor and ceiling bounds ensures rational sweepers across the full range of UTXOs, preventing exposure of dust to quantum adversaries while bounding payout for whale events.

### Why β = 2%?

The 2% rate is the value used in [1, §6.3] simulation analyses. It produces:

- Bounty-adjusted equilibrium `C^ss_new = r_f / [μ(1-β)] ≈ 21.45M RIN` at μ = 1.5%.
- Combined with γ_recovery = 10% (RIP-0006), `C^ss_ZKP ≈ 21.03M RIN` at μ = 1.7% (the Nakamoto target).

A higher β (e.g., 5%) over-incentivizes sweeping and creates excessive bounty volatility. A lower β (e.g., 0.5%) leaves too many sweep candidates economically unviable.

### Why k=5 for the floor multiplier?

The floor must exceed the gas cost by enough margin that:

- A sweeper has guaranteed positive expected value even after fee fluctuation.
- Competition among sweepers does not erode the bounty to zero (by escalating fees).

A factor of 5 is empirically reasonable: the fee market on Bitcoin and Litecoin shows fee-to-bounty ratios in this range historically. The factor is governable.

### Why a B_cap = 100,000 RIN ceiling?

100,000 RIN is the threshold above which a single sweep event becomes a macroeconomically significant liquidity injection. Bounding the per-event payout prevents:

- A sweeper of a 1M-RIN whale UTXO receiving 20,000 RIN (which would distort the market).
- MEV competition from concentrating on a small number of high-value events to the exclusion of broader sweep coverage.

The ceiling forces large-UTXO bounty efficiency to decline, encouraging sweepers to also process medium-value UTXOs.

### Why permissionless sweeping rather than auction?

An auction-based mechanism (where sweepers bid for the right to sweep specific UTXOs) would:

- Require an auction venue (centralization).
- Concentrate sweeping among a small set of operators.
- Create timing vulnerabilities (an auction's settlement window is itself a quantum-attack window).

Permissionless first-come-first-served sweeping with deterministic bounty computation is simpler, decentralized, and avoids auction venue capture.

### Alternatives considered

1. **Flat fixed bounty** (e.g., 1 RIN per sweep regardless of value) — Rejected. Either over-incentivizes dust sweeping or under-incentivizes whale sweeping.
2. **Logarithmic bounty function** — Rejected. More complex to specify and audit; piecewise-linear achieves the same goals with simpler arithmetic.
3. **Auction-based extraction** — Rejected (see above).
4. **Sweep batching (multiple UTXOs per transaction)** — Considered as a future optimization. Not specified in this RIP because it complicates the bounty calculation; future RIPs MAY add batched-sweep transaction types.

## Backwards Compatibility

This RIP defines a **hard fork** activating at block 15,768,000.

- Implementations not carrying this RIP's logic will reject sweep transactions as non-standard (the destination address `CV_ADDRESS` or pre-Vault address has no spend script under standard validation).
- Conversely, implementations carrying this RIP will reject blocks where post-τ_c UTXOs remain unswept after a grace period (TBD; specified in a future RIP if enforcement is desired). For the present specification, sweeping is **economically incentivized but not consensus-mandated**: a UTXO past τ_c that is never swept simply remains in the active set, exposed to whatever cryptographic threats exist. This is a deliberate non-mandate.

## Reference Implementation

A complete reference implementation is **NOT YET AVAILABLE**. The bounty function and validation logic are simple enough to admit a complete reference implementation well in advance of activation:

```cpp
// Pseudocode for sweep transaction validation
bool ValidateSweepTransaction(const CTransaction& tx, int nHeight, const Consensus::Params& params)
{
    // Identify sweep candidates among inputs
    for (const auto& input : tx.vin) {
        const Coin& coin = view.GetCoin(input.prevout);
        int dormancy = nHeight - coin.nHeight;
        if (dormancy < params.tau_c_blocks) {
            // Not a sweep candidate; standard validation applies
            continue;
        }

        // Compute bounty
        CAmount bounty = ComputeBounty(coin.out.nValue, params);
        CAmount residual = coin.out.nValue - bounty;

        // Verify outputs structure
        if (!OutputMatches(tx.vout[0], bounty, sweeper_address)) return false;
        if (!OutputMatches(tx.vout[1], residual, params.cv_address)) return false;
    }
    return true;
}

CAmount ComputeBounty(CAmount v, const Consensus::Params& params)
{
    CAmount linear = (v * params.beta_basis_points) / 10000;
    CAmount floor = params.k_floor * params.min_gas_cost;
    CAmount ceiling = params.bounty_cap;
    CAmount result = std::max(floor, std::min(linear, ceiling));
    if (result >= v) result = v - 1;  // dust safety
    return result;
}
```

(The "assumed floor" entry presumes a `min_gas_cost` of 5,000 satoshis and `k=5`, yielding `floor = 25,000 satoshis`.)

Additional consensus-level test vectors:

- TV-1: A sweep tx with `bounty + residual ≠ v` is rejected.
- TV-2: A sweep tx whose UTXO has `dormancy < τ_c` is rejected (not yet eligible).
- TV-3: A sweep tx whose `residual` is sent anywhere other than `CV_ADDRESS` is rejected.
- TV-4: Two competing sweep txs in the same block referencing the same UTXO: only the first (by block transaction order) is valid; the second is rejected as double-spend.
- TV-5: A sweep tx with bounty exceeding the ceiling is rejected.

## Security Considerations

### MEV competition

Sweep transactions are subject to ordering competition. Multiple sweepers may attempt to sweep the same high-value UTXO simultaneously, with the winner determined by block inclusion. This is structurally bounded MEV: the value at stake is the bounty (capped at `bounty_ceiling`), not the underlying UTXO value (which always goes to the CV regardless of which sweeper "wins").

The bounty ceiling (`B_cap = 100,000 RIN`) bounds the maximum incentive for ordering manipulation and limits the fee-bidding war that could otherwise consume the entire bounty.

### Adversarial sweeping

If an adversary breaks secp256k1, they can construct valid signatures for vulnerable UTXOs. Under this RIP, even an adversary with broken keys can only:

- Receive the bounty fraction (capped at 2% / B_cap).
- Direct the residual to themselves? **No.** The consensus rule mandates the residual destination is `CV_ADDRESS`. An adversary's "sweep" is structurally identical to a legitimate sweep: they receive only the bounty.

This is the central security property: **the CV cannot be bypassed even by an adversary with full key compromise**.

### Sweeper bounty depletion attack

An adversary cannot drain the network via the bounty: each bounty payout corresponds to an equivalent UTXO that crossed τ_c. The total bounty paid over time is bounded by `β × (cumulative entropic loss)`, which under nominal parameters is a tiny fraction of total supply per year.

### Fee-market interaction

Sweeper bounties create a small, predictable revenue stream for sweepers. This is conceptually similar to mining: sweepers compete on cost (gas efficiency) and possibly hardware (transaction propagation latency). The fee market may evolve to include "sweep priority" lanes; this is acceptable and does not require consensus intervention.

### Front-running of legitimate spends

If a legitimate owner spends their UTXO at dormancy = τ_c - 1 block, and a sweeper observes the mempool transaction and attempts to front-run with a sweep at τ_c, the sweep would be invalid (UTXO has been consumed by the legitimate spend). The mempool and standard double-spend rules prevent this attack.

### Pre-Vault address as honeypot

During years 30–444, the pre-Vault address aggregates significant value. An adversary who could break the consensus rule mandating its no-spend semantics could in principle drain it. Mitigations:

- The no-spend semantics are encoded as consensus rules (not as a "burn" address), with full validator enforcement.
- Implementations MUST reject any block containing a transaction with `vin` referencing the pre-Vault address, except for the specific PoR-activation migration transaction.
- Audit and review of the pre-Vault address's consensus enforcement is critical.
