## Abstract

This RIP integrates the **MimbleWimble Extension Block (MWEB)** privacy layer into Rincoin Core and specifies a consensus-level patch to the initial **Hogwarts Express (HogEx) integration transaction** that prevents `bad-txns-vin-empty` rejection at MWEB activation.

The MWEB design is inherited from Litecoin's LIP-0002 and LIP-0003 (David Burkett, 2019–2022) with adaptations for Rincoin's RinHash Proof-of-Work algorithm and 60-second block time. Full lifecycle support (Peg-in, Peg-out, blockchain reorganization resilience) is validated against the `rincoin-sim` regtest harness.

**Operational status (mainnet)**: The author has chosen to mark mainnet activation as BIP9 `NEVER_ACTIVE` pending a future successor RIP. The motivation is deferral until after the planned wallet migration from Berkeley DB 4.8 to SQLite descriptor wallets has completed. This RIP preserves the full specification so that activation can be re-proposed via a successor RIP without re-deriving the protocol design. See §2.1 for the rationale.

## Motivation

Privacy-preserving transaction layers have become a baseline expectation for sovereign monetary networks. Among the available options, MWEB provides:

- **No master keys or trusted setup**: Uses Pedersen commitments and confidential transactions natively.
- **Soft-fork compatibility**: Non-upgraded nodes continue to validate the transparent base layer without disruption.
- **Cut-through compaction**: Reduces long-term storage cost of the privacy layer compared to alternatives.
- **Production-validated**: Litecoin's MWEB has been live on mainnet since block 2,257,920 (May 2022) without consensus failures.

However, MWEB activation introduces a non-trivial consensus risk: the **HogEx transaction**, which serves as the integration point between the transparent base block and the MWEB extension block, has historically triggered `bad-txns-vin-empty` rejection in implementations that apply standard transaction-input validation rules to it. This was patched in Litecoin's `litecoind` v0.21 release; Rincoin's fork inherits the same vulnerability and requires the same patch.

This RIP specifies (a) the MWEB integration parameters for Rincoin and (b) the HogEx consensus fix.

## Specification

### 1. MWEB protocol inheritance

The MWEB transaction format, peg-in/peg-out semantics, kernel structure, MMR (Merkle Mountain Range) commitments, and cut-through rules are inherited verbatim from Litecoin LIP-0002 [1] and LIP-0003 [2]. This RIP does NOT redefine those structures; conforming implementations MUST adhere to the LIP-0002/LIP-0003 specifications.

The integration points specific to Rincoin are defined in the subsections below.

### 2.1 Strategic suspension on mainnet

The author has chosen to suspend MWEB activation on mainnet by setting its BIP9 deployment status to `NEVER_ACTIVE`. The choice is documented here as part of the specification so that a future successor RIP can reactivate from a known baseline.

#### Rationale

1. **Wallet-migration debt**: Rincoin Core currently uses Berkeley DB 4.8 (BDB 4.8) for `wallet.dat`. The project intends to migrate to SQLite descriptor wallets, following the trajectory established by Bitcoin Core (since v0.21) and being completed by Litecoin. If MWEB were activated on mainnet under the BDB 4.8 wallet format, MWEB-specific key material (Pedersen commitment scalars, view/spend keys, output blinding factors) would be persisted in BDB-format wallets. Subsequent migration to SQLite would then require an MWEB-aware migration path that handles confidential-output state across two wallet formats — a substantially harder problem than transparent-only migration.

2. **Wait for upstream completion**: Litecoin is in the process of completing SQLite descriptor-wallet integration with MWEB support. Postponing Rincoin's mainnet MWEB activation until that integration is well-tested upstream allows Rincoin to inherit the wallet-migration design rather than develop a parallel solution.

3. **Priority on security and sustainability updates**: The protocol's security and long-run sustainability layers — RIN3 hardening and the Customized Halving schedule (RIP-0002) — take precedence over the privacy layer in the current development phase. Activating MWEB now would consume engineering resources better directed toward those layers.

#### Implementation

- **Mainnet (`CMainParams`)**: The MWEB deployment in `consensus.vDeployments[Consensus::DEPLOYMENT_MWEB]` MUST be configured with `nStartTime = NEVER_ACTIVE` and `nTimeout = NEVER_ACTIVE` per BIP9 conventions. Conforming Rincoin Core builds for mainnet MUST NOT signal bit 4 nor enforce the MWEB consensus rules on mainnet blocks.
- **Testnet (`CTestNetParams`)**: MWEB is configured to activate at block 840, allowing end-to-end testing of the integration without mainnet exposure.
- **Regtest (`CRegTestParams`)**: MWEB activates at block 840 to match the `rincoin-sim` 1/1000-scale validation harness.

#### Conditions for re-proposal

This RIP does not specify the conditions under which MWEB SHOULD be reactivated; that determination belongs to a successor RIP. However, the following criteria are RECOMMENDED as preconditions for any successor RIP that proposes to lift the suspension:

- SQLite descriptor-wallet migration in Rincoin Core is complete and stable for at least one full release cycle.
- Litecoin's SQLite + MWEB integration has been live on Litecoin mainnet for at least 6 months without consensus or wallet-data incidents.
- Independent audit of the wallet-migration code path (BDB → SQLite, including MWEB state handling) has been published.
- Active mainnet testnet trials of MWEB on Rincoin's own testnet have validated the full Peg-in / Peg-out / Reorg / wallet-migration lifecycle.

A successor RIP that satisfies these preconditions MAY supersede this RIP via the standard RIP `Replaces` / `Superseded-By` mechanism defined in RIP-0001.

### 3. Address formats

Two distinct address formats coexist post-activation:

- **Transparent (Base58/Bech32)**: Existing Rincoin addresses. Format: `rrin1...` (Bech32, identical encoding to Litecoin's `ltc1...` with HRP `rrin`).
- **MWEB (Bech32m)**: New addresses. Format: `rrmweb1...` (HRP `rrmweb`, version byte 0).

MWEB addresses MUST NOT be valid as outputs of standard transparent transactions. They are valid only as recipients of peg-in transactions or internal MWEB-to-MWEB transfers.

### 4. HogEx consensus fix

The HogEx (Hogwarts Express) transaction is the special transaction that serves as the integration commitment between the transparent block and the MWEB extension block. It appears as the **last transaction in the transparent block body** when an MWEB extension block is present.

#### 4.1 The bug

A naïve port of pre-MWEB transaction validation rules to a Rincoin Core daemon will reject the initial HogEx transaction at MWEB activation height with error `bad-txns-vin-empty`. This occurs because:

- The HogEx transaction's `vin` may legitimately contain only MWEB-originated peg-out inputs that, in the transparent layer's view, look like a transaction with no traditional UTXO inputs.
- Stock validation logic enforces `vin.size() > 0` for all transactions except the coinbase.
- The HogEx is not a coinbase but legitimately may have an "empty" `vin` from the transparent layer's perspective.

#### 4.2 The patch

Validation logic MUST distinguish HogEx transactions and apply HogEx-specific input validation rather than generic non-coinbase rules. Conforming implementations MUST:

```
IsHogExTransaction(tx, block):
    return tx == block.vtx.back()                     // last tx in block
        && block.HasMWEBData()                        // block carries MWEB extension
        && tx.HogExStructureMarker() == HOGEX_MARKER  // structural identification
        // (See LIP-0002 for HOGEX_MARKER specification)

ValidateNonCoinbaseInputs(tx, block):
    if IsHogExTransaction(tx, block):
        return ValidateHogExInputs(tx, block)         // MWEB-aware validation
    if tx.vin.empty():
        return REJECT_INVALID, "bad-txns-vin-empty"
    // ... standard validation continues ...
```

The function `ValidateHogExInputs` MUST verify that:

- The HogEx `vin` correctly references the prior MWEB extension block's commitment.
- All peg-out outputs sum to exactly the value claimed in the MWEB extension block's peg-out kernel.
- The HogEx transaction itself contributes zero net value to the transparent layer's UTXO set (peg-ins exit the transparent set; peg-outs enter it; HogEx is the accounting bridge).

The full validation specification is in LIP-0002, §"HogEx".

#### 4.3 First-block edge case

At the activation block specifically, the prior block does not contain an MWEB extension. The HogEx transaction at the activation block MUST therefore be valid even when the prior MWEB state is empty (genesis-like state for the MWEB extension chain). This edge case is the specific source of the historic `bad-txns-vin-empty` failure on first activation.

The patch MUST be applied to all conforming implementations before block 432,000.

### 5. Peg-in transaction

A peg-in transaction:

1. Originates from a transparent address.
2. Contains exactly one output of type `OP_MWEB_PEGIN <commitment>`.
3. Has its value locked in the MWEB extension block via the HogEx accounting.

```
PEG-IN TX (transparent → MWEB):
  vin:  standard transparent UTXO inputs
  vout: [OP_MWEB_PEGIN <commitment>]
        [optional change output, transparent]
```

### 6. Peg-out transaction

A peg-out transaction:

1. Originates from an MWEB address (signed via Pedersen commitment).
2. Produces an output spendable on the transparent layer.
3. Is finalized by the HogEx of the next mined block.

A 6-block confirmation depth is REQUIRED for peg-out maturity.

### 7. Reorganization handling

On a chain reorganization that invalidates a block containing peg-in or peg-out transactions:

- The transparent layer rolls back per standard reorg rules.
- The MWEB MMR commitments roll back to the equivalent pre-fork state.
- Peg-in commitments not yet referenced in any kernel are returned to mempool.
- Peg-out outputs that were spent on the post-fork chain are returned to MWEB UTXO set.

The full reorg semantics are inherited from LIP-0002 and validated in `rincoin-sim` (see Test Vectors below).

### 8. Mining policy

Mining (`generatetoaddress`) MUST direct the coinbase output to a transparent address. MWEB addresses are NOT valid coinbase recipients. This matches the Litecoin MWEB convention and is enforced at the consensus layer.

## Rationale

### Why inherit from Litecoin rather than reimplement?

MWEB is a complex consensus mechanism with subtle privacy properties. A from-scratch reimplementation would require independent cryptographic auditing of the Pedersen commitment scheme, range proof construction, kernel offset arithmetic, and MMR consistency proofs. Litecoin's MWEB has been deployed on mainnet for ~4 years (2022–2026) without privacy-affecting bugs, providing strong empirical evidence of correctness.

Inheriting LIP-0002/LIP-0003 verbatim and patching only the integration points is the lower-risk path. This is the same strategy used by other Litecoin-derived chains.

### Why soft fork rather than hard fork?

The MWEB extension block design is intentionally backwards-compatible: non-upgraded nodes see only the transparent block, ignore the MWEB extension data, and continue to validate normally. This is a strict feature-add, not a rule change. Soft-fork activation imposes the lowest coordination cost on full-node operators.

### Why 75% signaling threshold (vs. 90% for RIP-0003)?

MWEB activation is consensus-additive (it adds a new transaction type without changing existing validation rules), whereas RIP-0003's CSV is consensus-modifying (it changes the subsidy schedule). The lower threshold reflects the lower coordination risk of an additive change. 75% matches Litecoin's MWEB activation threshold.

### Why activation was originally proposed at block 432,000?

This corresponds to the activation height tested in `rincoin-sim` at 1/1000 scale (block 432). The position within Phase 2 (block range 420,000–629,999) was chosen to provide:

- Sufficient post-genesis maturation (~10 months) for the network to stabilize before introducing privacy infrastructure.
- A subsidy environment (12.5 RIN/block) substantial enough to incentivize rapid signaling.
- Distance from the Phase 3 → 4 boundary (block 840,000) so that MWEB activation would not collide with the Customized Halving consensus event.

This rationale is preserved as historical context. As of this RIP's current revision, the mainnet activation has been suspended via `NEVER_ACTIVE` per §2.1; any successor RIP proposing reactivation will need to select a fresh block height appropriate to network state at that time.

## Backwards Compatibility

This RIP defines a **soft fork** that is suspended on mainnet via `NEVER_ACTIVE` (§2.1). Consequences:

- **Mainnet (current state)**: MWEB consensus rules are NOT enforced. Mainnet blocks containing MWEB extension data would be rejected as malformed. Mainnet wallets MUST NOT generate MWEB-format addresses (`rrmweb1...`); attempts to send to such addresses MUST fail at the wallet layer with a clear error indicating mainnet MWEB is suspended.
- **Testnet and regtest**: The soft fork activates as specified in §2 and behaves per the rules below.

For the testnet/regtest activated state, the soft-fork compatibility model is:

- **Non-upgraded nodes**: Continue to validate the transparent layer correctly. They will NOT see MWEB transactions and will NOT enforce the HogEx fix. They MAY observe peg-in transactions as outputs to "unspendable" `OP_MWEB_PEGIN` scripts; these are still valid transparent transactions with valid signatures, just with outputs that the upgraded layer interprets specially.
- **Upgraded nodes**: Validate both layers. Will reject blocks where the HogEx is malformed or where MWEB extension data is inconsistent with HogEx commitments.
- **First-block hazard (historical, not currently in force on mainnet)**: The historic `bad-txns-vin-empty` rejection required that the HogEx fix be deployed in 100% of mining nodes before any mainnet activation block. This hazard does not apply under the current `NEVER_ACTIVE` state, but a successor RIP that proposes reactivation MUST address it explicitly.

## Test Vectors

The following test cases MUST pass on any conforming implementation. All have been validated on `rincoin-sim` v1.0.6.1 (test date 2026-04-25).

### TV-1: Peg-In

```
GIVEN: Wallet with 13,975 RIN at transparent address rrin1q8twvefee...
WHEN:  sendtoaddress("rrmweb1qqfd6u6nk4pvyu9...", 10)
       generatetoaddress(1)
THEN:
  - Block N+1 contains 2 transactions: coinbase + peg-in TX
  - Peg-in TX outputs: [OP_MWEB_PEGIN <commitment>, transparent change ~13,975 RIN minus fees]
  - Block N+1 contains MWEB extension block with new commitment
  - HogEx transaction in block N+1 validates successfully
  - listaddressgroupings shows: rrmweb1qqfd6u6nk4pvyu9... = 10.00000000 RIN
  - listaddressgroupings shows: rrmweb1qq0mdeg9msd2hqm... = 14.99963500 RIN (auto-generated change)
```

### TV-2: Peg-Out

```
GIVEN: MWEB balance of 10 RIN at rrmweb1qqfd6u6nk4pvyu9...
WHEN:  Send 5 RIN from MWEB to a fresh transparent address rrin1...
       generatetoaddress(1)
THEN:
  - Transparent recipient receives exactly 5.00000000 RIN
  - MWEB remaining balance ≈ 4.99991600 RIN (5.0 minus MWEB transaction fee)
  - HogEx transaction reflects net 5.0 RIN peg-out
```

### TV-3: Reorg Resilience

```
GIVEN: Confirmed peg-out transaction in block at height 451
WHEN:  invalidateblock(<hash of block 451>)
       Generate a dummy transaction  // ensure unique block hash
       generatetoaddress(2)            // re-mine to surpass invalidated tip
THEN:
  - After invalidate: chain tip height = 451 (one less than original 452)
  - Original peg-out TX status: 0 confirmations OR orphaned
  - After re-mine: chain tip height = 452 (matches original)
  - MWEB MMR state is consistent with re-mined block sequence
  - No double-spend of peg-out funds detected
```

### TV-4: First-block HogEx (consensus fix verification)

```
GIVEN: Chain at activation height - 1 (no prior MWEB extension)
WHEN:  Mine the first block at activation height with a HogEx referencing empty prior MWEB state
THEN:
  - Block validates successfully
  - HogEx transaction does NOT trigger bad-txns-vin-empty
  - MWEB extension block is initialized with empty UTXO set and zero kernel offset
```

### TV-5: HogEx malformation rejection

```
GIVEN: Chain post-activation
WHEN:  Construct a block with malformed HogEx (e.g., peg-out total ≠ MWEB peg-out kernel claim)
THEN:
  - Block is rejected with consensus error indicating HogEx accounting mismatch
  - Chain tip does NOT advance
```

## Security Considerations

### Privacy properties

MWEB provides:

- **Confidential amounts**: Pedersen commitments hide transaction values.
- **Cut-through**: Intermediate transactions can be compacted, reducing the linkability signal.
- **Address blinding**: MWEB addresses are not directly linkable to transparent identities without explicit peg-in/peg-out reference.

MWEB does NOT provide:

- **Sender anonymity at peg-in**: A peg-in transaction is publicly visible on the transparent layer with the source transparent address.
- **Recipient anonymity at peg-out**: Similarly, the destination transparent address is visible.
- **Network-level privacy**: MWEB transactions traverse the standard P2P network with no Dandelion or Tor integration in this RIP. Network-level privacy is out of scope for this RIP and may be addressed by a future RIP.

Users MUST understand that MWEB protects in-flight value but not the perimeter (peg events).

### Activation hazard window

Under the current `NEVER_ACTIVE` mainnet configuration (§2.1), the activation hazard does not apply on mainnet. For testnet and regtest, the original concern remains: if soft-fork signaling fails to reach the threshold by the timeout, activation does not occur; if it reaches threshold but a non-trivial minority of mining hash power has not deployed the HogEx fix, the activation block is at risk of producing chain splits. Coordinated deployment is the operational responsibility of mining pool operators on those networks.

A successor RIP that proposes lifting the `NEVER_ACTIVE` suspension on mainnet MUST address the activation hazard window explicitly, including specifying mining-pool readiness verification before the activation block is reached.

### Reorg double-spend window

During a reorg that affects peg events, there is a brief window in which a peg-out output may be spent on both the original and the re-mined chain (until the HogEx-level accounting reconciles). The 6-block peg-out maturity requirement is intended to ensure this window is closed before peg-out funds are usable.

### Cryptographic dependencies

MWEB relies on the discrete logarithm assumption over secp256k1 (for Pedersen commitments) and on the soundness of the bulletproof range proof scheme. These dependencies are inherited from Litecoin's MWEB and from the broader Bitcoin cryptographic stack. A break of either would compromise both Rincoin's MWEB and a substantial portion of the cryptocurrency ecosystem.

### Legacy address quarantine interaction

When the Cryptographic Vault and Phased Legacy Address Migration mechanisms (RIP-0006, RIP-0008) eventually deprecate legacy elliptic curve cryptography, MWEB outputs MUST also undergo the migration. The migration of MWEB UTXOs is non-trivial because the Pedersen commitments themselves rely on EC primitives. Detailed handling is deferred to RIP-0008 and any future MWEB-specific migration RIP.
