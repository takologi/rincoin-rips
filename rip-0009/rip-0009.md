## Abstract

This RIP specifies a transaction-version replay-protection mechanism for Rincoin. From the RinHash fork height (Block 840,000 on mainnet), every non-exempt transaction included in a block MUST carry `nVersion = 0x52494e33` (ASCII "RIN3", big-endian = 1,380,535,859). The marker version embeds RinHash identity into every transaction's sighash preimage, providing single-byte fork-id-style replay protection against pre-fork and sibling chains without altering the sighash algorithm.

Enforcement is applied across three independent layers — consensus, mempool admission, and block assembly — to close a denial-of-service vector not addressed by the reference implementation.

---

## Motivation

Rincoin descends from Bitcoin → Litecoin → Rincoin. Transactions valid on a pre-fork chain (or on a chain that shares Rincoin's UTXO history) could otherwise be replayed on the post-fork chain. Embedding a distinct `nVersion` marker into the signed transaction body causes any signature produced for the post-fork chain to be invalid on chains that do not require the marker, and vice versa, achieving replay protection with a minimal, well-understood change surface.

## Specification

### 1. Constant

```cpp
// src/primitives/transaction.h
static const int32_t RIN_FORK_TX_VERSION = 0x52494e33; // "RIN3" BE = 1380535859
```

The decimal value is **1,380,535,859**. Implementations MUST use this exact value; an incorrect constant produces sighash divergence and a permanent chain split.

---

### 2. Activation Height

| Network | `nRinHashForkHeight` | Relation |
|---------|---------------------|----------|
| Mainnet | 840,000 | 4 × `nSubsidyHalvingInterval` (210,000) |
| Testnet | 840 | 4 × 210 |
| Regtest | 840 | 4 × 210 |

RIN3 activates at the **same height as the Customized Halving hard fork (RIP-0002)**. Consolidating all consensus changes into a single fork event at Block 840,000 minimizes operational disruption for node operators and mining pools.

---

### 3. Enforcement Layers

A transaction is *exempt* if it is a coinbase, a HogEx transaction, or an MWEB-only transaction. For every non-exempt transaction at effective height `h >= nRinHashForkHeight`, `nVersion` MUST equal `RIN_FORK_TX_VERSION`.

**Layer 1 — Consensus (`ContextualCheckBlock`, `validation.cpp`):** A block containing any non-exempt transaction whose `nVersion` is not `RIN_FORK_TX_VERSION` at `h >= nRinHashForkHeight` is rejected with `bad-tx-rinhash-version`. This is the authoritative consensus rule.

**Layer 2 — Mempool admission (`MemPoolAccept::PreChecks`, `validation.cpp`):** A transaction submitted via `sendrawtransaction` or relayed to the mempool, where the next block height (`ChainActive().Tip()->nHeight + 1`) is `>= nRinHashForkHeight`, is rejected with `TX_CONSENSUS` / `bad-tx-rinhash-version` if it is non-exempt and lacks the marker version. MWEBOnly is exempt, consistent with Layer 1.

**Layer 3 — Block assembly (`BlockAssembler::TestPackageTransactions`, `miner.cpp`):** Belt-and-suspenders skip. If a legacy-`nVersion` transaction reaches the block-assembly package despite Layer 2, it is skipped (`return false`) so `CreateNewBlock` never emits an invalid block.

---

### 4. Wallet Behaviour

`wallet/txassembler` sets the outgoing transaction `nVersion` based on the chain tip at construction time:

```
if (last_height >= nRinHashForkHeight - 1)  // = 839 on regtest
    nVersion = RIN_FORK_TX_VERSION;
else
    nVersion = legacy;
```

The `- 1` offset ensures a transaction constructed while the tip is at `nRinHashForkHeight - 1` is already marker-versioned, so that it remains valid when mined into the fork block itself, avoiding a race at the boundary.

## Rationale

### Improvements Over the Reference Implementation

1. **Mempool Zombie DoS prevention (original).** The reference describes only consensus-layer enforcement and relay standardness. It does not address a denial-of-service vector in which a high-fee legacy-`nVersion` transaction is admitted to the mempool, is then selected by the block assembler, and causes `TestBlockValidity` to throw inside `CreateNewBlock` — stalling block production for as long as the transaction lingers (up to the mempool expiry). Layer 2 closes this by performing a height-aware rejection at mempool entry; Layer 3 provides defence in depth.

2. **CH-integrated single hard fork (original).** The reference positions its era at Block 630,000 (3 × halving interval). This RIP positions RIN3 at Block 840,000 (4 × halving interval) so that it activates simultaneously with the Customized Halving dilation (RIP-0002) and the minimum-peer-version floor, presenting node operators and pools with a single, well-announced fork event.

3. **Scope reduction (governance decision).** The reference introduces a general init+overlay era table covering PoW parameters, a coinbase OP_RETURN marker, the fork tx version, and a peer-protocol floor, driven by a generated C++ header from JSON. The Core Authority evaluated the full era system and classified it as over-engineering for current needs. This RIP extracts only the RIN3 nVersion rule; the OP_RETURN `RIN1` branding marker is deferred (optional, no consensus impact) and the minimum peer version 70018 is tracked separately as a v1.1.0 networking change.

4. **Attack-simulation verification (original).** Replay-protection correctness and reorg resilience are demonstrated by dedicated simulation harnesses up to a 5,461-block reorg that erases the entire post-CH history. See the Verification section.

---

## Backwards Compatibility

This is a hard fork. Before `nRinHashForkHeight` the rule is dormant and historical blocks validate unchanged. Wallets and services that construct raw transactions MUST set `nVersion = 1380535859` for transactions intended for inclusion at or after the fork height. Coinbase, HogEx, and MWEB-only transactions are unaffected.

Nodes that do not upgrade will reject the post-fork chain (or fail to produce acceptable blocks) and diverge at Block 840,000.

## Security Considerations

- **Incorrect constant.** Using a wrong decimal for `RIN_FORK_TX_VERSION`
  produces sighash divergence on every post-fork transaction, causing a
  permanent chain split. Implementations MUST verify the constant is exactly
  `0x52494e33` = 1,380,535,859 and MUST include a test vector confirming
  this before deployment.

- **Mempool Zombie DoS.** Without Layer 2, a high-fee legacy-nVersion
  transaction stalls `CreateNewBlock` until the 14-day mempool expiry,
  denying block production. Layer 2 (height-aware mempool rejection) and
  Layer 3 (block-assembly skip) together eliminate this vector. Operators
  MUST deploy both layers; deploying only Layer 1 leaves the DoS window open.

- **Boundary race condition.** A transaction constructed at tip height
  `nRinHashForkHeight - 1` must already carry `RIN_FORK_TX_VERSION`, as it
  may be confirmed in the fork block itself. The `last_height >= fork - 1`
  offset in `txassembler` handles this. Wallet implementations that compute
  the version based on the *next* block height (rather than the tip) MUST
  apply the equivalent guard.

- **Exempt transaction abuse.** The coinbase, HogEx, and MWEBOnly exemptions
  are enforced at all three layers using identical predicate logic. Any
  divergence between layers in classifying a transaction as exempt could allow
  a transaction to pass mempool admission but be rejected at consensus,
  causing node inconsistency. Implementations MUST use the same exemption
  predicate at all three enforcement points.
