## Abstract

This RIP specifies the **Proof of Rinne (PoR) Reincarnation Consensus Mechanism**, the regenerative core of the Rincoin protocol. PoR transforms passive entropic loss (irretrievable assets due to lost private keys, hardware degradation, and similar Type A attrition) from a permanent deflationary sink into a regenerative security budget by recycling these dormant assets into new block subsidies after a deterministic dormancy threshold.

PoR establishes a Dual-Layer Temporal Architecture: a Transactional Layer (Layer 1) handling active value transfer, and a Rinne Layer governed by Proof-of-Temporal-History (PoTH) that manages the thermodynamic lifespan of dormant assets. The protocol enforces a Thermodynamic Statute of Limitations (τ ≈ 400 years) on truly dormant addresses, after which the assets are combusted and re-emitted as block rewards on the Transactional Layer.

This RIP is a **long-horizon specification**: full activation occurs at block height ≈ 233,280,000 (~444 years post-genesis at 60 s/block). The activation is automatic, conditional on chain progression past `t_max`, and requires no signaling. This RIP defines the consensus rules; the parameter governance procedures for the delay parameter τ are also specified here.

## Motivation

### The exhaustion problem

Per RIP-0002, Rincoin's deterministic emission schedule asymptotes to a perpetual block subsidy of 0.6 RIN. The cumulative supply curve approaches 168,000,000 RIN as t → ∞, but the *circulating supply* (after entropic loss) approaches a steady state `C^ss = r_f / γ ≈ 20.87M RIN` and then decays toward zero on a multi-century timescale if entropic loss exceeds new issuance.

Without intervention, after the cessation of mining (`t_max ≈ 444` years post-genesis under Scenario II), the network has no mechanism to replenish lost capital. Over a secular horizon, the circulating supply continues to decay at γ ≈ 1.5%/yr, and the security budget for the Proof-of-Work base layer must come entirely from transaction fees. Empirical evidence from Bitcoin suggests fee-only security is uncertain at any plausible scale.

### The conservation insight

Crucially, lost coins are not destroyed—they are merely inaccessible. They remain on the ledger, distinguishable only by their dormancy. If the protocol can identify dormant assets and reclaim them on a sufficiently long timescale (long enough that legitimate cold-storage holders are not penalized), the entropy can be recycled rather than wasted.

This is the regenerative principle: passive entropic decay becomes a fuel source for perpetual security.

### The Tetra-Lemma resolution

PoR completes the protocol's resolution of the Blockchain Tetra-Lemma ([1, §1.3])—the four-dimensional optimization across decentralization, security, scalability, and thermodynamic sustainability—by closing the economic loop. Without PoR, sustainability is sacrificed. With PoR, all four dimensions are preserved without compromise.

## Specification

### 1. Dual-Layer Architecture

Post-activation, the Rincoin protocol operates two consensus-coordinated layers:

#### Layer 1: Transactional Layer (the existing chain)

The standard Rincoin blockchain, handling all active value transfers, smart contracts (when introduced by future RIPs), and standard transactions. Block production and consensus on this layer continue under RIP-0002's emission schedule.

#### Layer 2: Rinne Layer (Rinnechain)

A specialized temporal layer that does NOT process transactions in the conventional sense. The Rinnechain produces blocks that:

- Witness the dormancy state of all Layer-1 UTXOs.
- Compute the Cumulative Loss Function `L(t)` and the Rinnechain Reserve `R_in(t)`.
- Trigger Reincarnation Events when dormant assets reach maturity.

The Rinnechain has a target block time of 144,000 Layer-1 blocks (~100 days) and uses Proof-of-Temporal-History (PoTH), described below.

### 2. Proof of Temporal History (PoTH)

PoTH is a witness-and-attestation consensus protocol where each Rinnechain block:

1. References a specific Layer-1 block height `h_attest`.
2. Includes a Merkle commitment to the set of UTXOs at `h_attest` that have been continuously unspent for at least τ Layer-1 blocks.
3. Identifies UTXOs newly crossing the τ-block dormancy threshold (the "Rinne candidates").
4. Provides Layer-1 chain-of-custody proofs for each Rinne candidate (proving non-spend across the full τ window).

PoTH validators are the same set of Layer-1 miners (no separate validator set is introduced in this RIP). A Rinnechain block is valid if and only if:

- Its `h_attest` is at least 144,000 Layer-1 blocks past the prior Rinnechain block's `h_attest`.
- Its Merkle root commits to a UTXO set provably consistent with Layer-1 chainstate.
- Its Rinne candidate set contains exactly the UTXOs whose dormancy-since-creation equals or exceeds τ at `h_attest` and which are not already in a prior Rinnechain block's candidate set.

### 3. Thermodynamic Statute of Limitations

A Layer-1 UTXO is designated a **Rinne candidate** when it satisfies:

```
last_spend_block(utxo) + τ_blocks <= current_block_height
```

where `τ_blocks` is the dormancy threshold in Layer-1 blocks.

#### Initial value of τ

```
τ_blocks_initial = 210,240,000 blocks  (≈ 400 years at 60 s/block)
```

This corresponds to τ ≈ 400 years, matching the value used in [1, §4.2.1] for analytical and simulation purposes.

#### τ governance

τ is a **governable parameter** subject to the procedure in §6 of this RIP. Changes to τ MUST satisfy:

- `τ_min = 157,680,000 blocks` (300 years), and
- `τ_max = 315,360,000 blocks` (600 years).

The bounds are normative consensus rules; a Rinnechain block proposing τ outside these bounds is invalid.

### 4. Reincarnation Event

Once a UTXO is included in a Rinnechain block as a confirmed Rinne candidate (see §5 below for confirmation rules), the protocol performs a **Reincarnation Event**:

1. **Combustion**: The UTXO is removed from the Layer-1 active UTXO set. Its value `v_combust` is recorded in the Rinnechain reserve `R_in`.
2. **Reincarnation budget allocation**: The combusted value contributes to a perpetual reincarnation reward stream.
3. **Block reward augmentation**: For all subsequent Layer-1 blocks during the reincarnation window, the block subsidy is increased by:

```
ΔSubsidy = r_re / B_y
```

where:

- `r_re = r_f` (the annual reincarnation rate matches the terminal subsidy rate, i.e., 315,360 RIN/yr under RIP-0002 baseline).
- `B_y = 525,600` blocks per year.
- Therefore `ΔSubsidy = 0.6 RIN/block` augmenting the terminal subsidy.

The reincarnation continues until `R_in` is depleted, at which point the protocol returns to the unaugmented terminal subsidy until the next Reincarnation Event triggers.

### 5. Confirmation rules

A Rinne candidate proposed in Rinnechain block `B_R` becomes **confirmed** (eligible for combustion) when:

- `B_R` has been buried under at least 6 subsequent Rinnechain blocks (≈ 600 days of additional dormancy verification), AND
- No Layer-1 spend of the candidate UTXO has occurred during this confirmation period.

If a Layer-1 spend is observed during the confirmation period, the candidate is REJECTED. The owner has reactivated the address, and the assets are not subject to the statute of limitations. Such a spend resets the dormancy clock to zero from the Layer-1 perspective.

This confirmation buffer is critical: it provides a final ~600-day window in which long-term holders (or their inheritors) can demonstrate liveness before assets are combusted.

### 6. Active Regeneration Phase

The mathematical evolution of the Rinnechain reserve `R_in(t)` is governed by the integro-differential identity in [1, §4.3.1]:

```
For t < τ:
    R_in(t) = 0    (no candidates have yet matured)

For τ ≤ t < t_re:
    R_in(t) = L(t - τ)    (cumulative loss matured to candidacy)

For t ≥ t_re:
    R_in(t) = R_in(t_re) + ∫[t_re, t] [γ·C_Rin(u-τ) - r_f] du
```

where:

- `t_re = t_max` is the recovery activation time (mining cessation under RIP-0002).
- `γ` is the continuous loss rate (≈ 0.0151 under Scenario II).
- `C_Rin(t)` is the effective circulating supply with reincarnation active.
- `L(t)` is the Cumulative Loss Function `∫[0, t] γ·C_CH(u) du`.

The detailed numerical implementation is provided in [2] (`rincoin-sim` simulation suite) and in the companion repository [3].

### 7. Global Conservation Law

At any instant `t ≥ t_re`, the theoretical maximum supply `S_max = 168,000,000 RIN` is identically equal to:

```
S_max ≈ C_Rin(t)                    (Active State)
      + R_in(t)                     (PoR Reserve)
      + ∫[t-τ, t] γ·C_Rin(u) du     (Pending Entropy / Latent State)
```

This is a **mandatory consensus invariant**: any chainstate that violates this identity beyond a tolerance of 1 satoshi MUST be rejected as invalid. Implementations MUST verify this identity at every Reincarnation Event.

## Delay Parameter Governance

This section integrates the procedures specified in [1, §5] for governance of τ.

### 6.1 Target equilibrium ratio

The protocol targets an asymptotic distribution of value across thermodynamic states:

```
R_in : Pending Entropy : C_Rin  ≈  77 : 70 : 21
```

normalized to `S_max = 168M RIN`. Deviations from this ratio MAY trigger τ adjustments per the governance procedures below.

### 6.2 Baseline stabilization (PID with dead-time compensation)

A discrete-time Proportional-Integral-Derivative controller with a Smith Predictor for dead-time compensation produces τ adjustments:

```
e(t) = R_in(t) / C_Rin(t) - r_target
Δτ_k = K_p · e(t_k) + K_i · Σ e(t_j)·Δt + K_d · (e(t_k) - e(t_k-1)) / Δt
```

with normative gain bounds:

- `K_p ∈ [0.001, 0.05]`
- `K_i ∈ [0.0001, 0.01]`
- `K_d ∈ [0.0001, 0.01]`

Initial gains: `K_p = 0.01, K_i = 0.001, K_d = 0.0005`.

### 6.3 Predictive optimization (MPC, optional)

Future RIPs MAY upgrade the controller to a Model Predictive Control formulation per [1, §5.2]. This is an extension point.

### 6.4 Adjustment cadence and bounds

τ adjustments are applied at the boundary of each Rinnechain epoch (every 144,000 Layer-1 blocks):

- Single-epoch adjustment magnitude: `|Δτ| ≤ 525,600 blocks` (≈ 1 year).
- Cumulative annual adjustment: `|Σ Δτ over 525,600 L1 blocks| ≤ 5,256,000 blocks` (≈ 10 years).

These rate limits prevent regulatory volatility and ensure smooth parameter evolution.

### 6.5 Stochastic robustness

τ adjustments operate on stochastic input data. The Rincoin Whitepaper [1, §5.4] augments the deterministic dynamics with a Wiener process:

```
dR_in(t) = [γ·C_Rin(t-τ) - r_f] dt + σ·dW_t
```

Implementations SHOULD apply Bayesian MCMC posterior estimation to τ adjustments to account for measurement noise. This is RECOMMENDED but not normative; the deterministic PID controller is the minimum conforming implementation.

## Rationale

### Why a 400-year statute of limitations?

The choice of τ ≈ 400 years balances two pressures:

1. **Too short**: penalizes legitimate long-horizon holders (estate planning, generational wealth transfer, deep cold storage).
2. **Too long**: insufficient regeneration to sustain the security budget on a multi-millennium timescale.

The whitepaper [1, §4.3] argues that 400 years exceeds any plausible single-generation cold-storage horizon (no individual or institution maintains active key rotation across 4+ generations) while being short enough that the asymptotic regeneration rate matches the terminal subsidy rate. Numerical results in [1, §4 and §6] show this choice produces the target equilibrium ratio of 77:70:21.

### Why no signaling activation?

PoR activation is **time-deterministic**: it activates automatically when chain progression passes `t_max ≈ 444 years post-genesis`. There is no possibility of "early" or "late" activation; the trigger is the cessation of subsidy under RIP-0002. Adding a signaling layer would introduce political risk (miners could refuse to signal) without corresponding benefit (the activation is a known consensus rule from genesis).

However, full nodes MUST carry the PoR consensus logic from network launch, even though it produces no observable behavior for the first ~444 years. This preserves the credible commitment that the regeneration mechanism will activate as scheduled.

### Why store the reserve on a separate Rinnechain?

The Rinne Layer is conceptually a distinct ledger because:

- It tracks a fundamentally different notion of state (dormancy duration) than the transactional ledger.
- Its block cadence (~100 days) is grossly mismatched with Layer 1's 60-second cadence.
- Embedding Rinnechain data into Layer-1 blocks would impose ongoing block-size overhead even when no Reincarnation Events are pending.

The two-layer design isolates the costs.

### Why are miners both Layer-1 and Layer-2 validators?

Introducing a separate validator set for the Rinne Layer would create governance friction. The existing miner set already has the cryptographic capability (Layer-1 chain access) and the economic incentive (the reincarnation budget directly augments their block reward) to validate PoTH. A separate validator set is unnecessary.

### Alternatives considered

1. **Tail emission (Monero-style)** — rejected per RIP-0002 rationale (violates macro-cap).
2. **Lockless inheritance (e.g., on-chain wills)** — rejected because it requires identity oracles or pre-registered beneficiaries, both of which violate the trustless model.
3. **Demurrage (continuous decay tax on all balances)** — rejected because it penalizes liveness, not dormancy.
4. **Shorter τ (e.g., 50 years)** — rejected because it intersects plausible single-lifetime cold-storage horizons.

## Backwards Compatibility

This RIP defines a **hard fork** that activates at `t_max ≈ 444 years post-genesis`. Until activation:

- Implementations not carrying PoR logic will produce identical chainstate to implementations that do (PoR is dormant).
- The Rinnechain produces no blocks during this period. Specifically, no Rinne candidates exist during the first τ years post-genesis (no UTXO has accumulated 400 years of dormancy yet).

Post-activation:

- Implementations not carrying PoR logic will fail to validate Rinnechain blocks or to enforce the Global Conservation Law. They will diverge from the PoR-aware chain at the first Reincarnation Event.

### Carrying the dormant logic

Mainnet launch implementations MUST include the PoR consensus rules even though they will not trigger for ~444 years. This is necessary because:

- Adding the rules later would itself be a hard fork requiring coordination across the entire network.
- Pre-committing to the rules at genesis is the credible commitment the protocol's regenerative claims rely upon.

## Security Considerations

### Liveness incentive

PoR creates a strong incentive for any UTXO holder to make at least one spend (even self-to-self) within τ blocks of any prior spend. This is a positive design feature: it discourages permanent loss of access. However, it also means that adversaries who briefly compromise a key may be incentivized to delay their attack until close to the τ deadline (to maximize the window before recovery).

This is partially mitigated by the 6-block (~600-day) confirmation buffer in §5: even after candidacy, the legitimate owner has additional time to demonstrate liveness.

### Long-time-horizon adversarial models

The 400-year horizon is long enough that the cryptographic primitives in use today may be broken. Specifically:

- secp256k1 (used for transparent address signatures) is presumed quantum-vulnerable within 30–100 years.
- SHA-256 (used in PoW and address hashing) may face cryptanalytic improvements.

The Cryptographic Vault (RIP-0006) addresses the quantum vulnerability by quarantining vulnerable UTXOs into a keyless smart contract before assets reach PoR maturity. PoR itself does not directly address cryptographic obsolescence; it interfaces with the Vault to receive maturity-graded assets.

### Reorg resistance

A reorg of more than τ blocks would invalidate Reincarnation Events. This is operationally impossible at the τ ≈ 210,240,000-block scale, but the protocol still SHOULD verify Reincarnation Event finality by requiring deep PoW commitment (consensus-enforced post-PoR-activation: Reincarnation Events at depth < 1,000 blocks SHOULD NOT be considered final by external observers).

### Conservation Law enforcement is critical

The Global Conservation Law (§7) is the central correctness invariant. A bug in its evaluator could allow either:

- Inflation (R_in over-credited, leading to over-issuance of reincarnation rewards), or
- Stuck assets (R_in under-credited, leading to permanent loss of legitimate dormant value).

Implementations MUST treat the Conservation Law evaluator as a security-critical component subject to the same audit standards as the consensus rules.

### τ governance attack surface

The PID controller for τ is parameterized by gains (`K_p, K_i, K_d`). If an attacker can manipulate the inputs to the controller (R_in / C_Rin ratio), they can drive τ toward bounds. Mitigations:

- The controller's `Δτ` per epoch is bounded.
- Multiple Rinnechain epochs (~years) of sustained manipulation are required to drive τ to either bound.
- Inputs are derived from on-chain state, not external oracles.
