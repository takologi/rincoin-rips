## Abstract

This RIP specifies the **Customized Halving Schedule** for Rincoin: a multi-phase block subsidy emission schedule that dilates the inter-halving interval after block 840,000 to maintain security budget continuity beyond the point at which a naïve Bitcoin-standard schedule would cause issuance to fall below the network's security threshold.

The schedule preserves Bitcoin-standard 210,000-block intervals for the first four halvings, transitions to extended intervals from block 840,000 onward, and converges to a perpetual terminal block reward of 0.6 RIN at block 6,300,000. The cumulative supply asymptotes to 168,000,000 RIN.

This RIP specifies **only the deterministic emission schedule**. The macroeconomic homeostasis claimed in the Rincoin Whitepaper [1] is a property of the combined system including the Proof of Rinne (PoR) reincarnation consensus, which is specified separately in RIP-0005. The conditional fallback emission schedule (Scenario III) is specified in RIP-0003.

## Motivation

Rincoin operates with a 60-second target block time, ten times faster than Bitcoin's 600-second target. If the standard 210,000-block halving interval were preserved without modification, the security budget contributed by block subsidies would degrade as follows:

| Halving | Block Height | Approx. Time | Reward (RIN) | Annual Issuance (RIN) |
|---|---|---|---|---|
| 0 | 0 | t = 0 | 50 | 26,280,000 |
| 1 | 210,000 | t ≈ 0.40 yr | 25 | 13,140,000 |
| 2 | 420,000 | t ≈ 0.80 yr | 12.5 | 6,570,000 |
| 3 | 630,000 | t ≈ 1.20 yr | 6.25 | 3,285,000 |
| 4 | 840,000 | t ≈ 1.60 yr | 3.125 | 1,642,500 |

Per the Rincoin Whitepaper [1, §1.2], the issuance rate falls below Bitcoin's initial generation rate (50 BTC per 600 seconds ≈ 4,380,000 BTC/yr) approximately 583 days after genesis. Without intervention, the security budget contracts ten times faster than Bitcoin's, while the fee market is unlikely to mature on a comparable accelerated timeline.

The Customized Halving Schedule resolves this dissonance by dilating the halving interval from block 840,000 onward, restoring an effective four-year economic cycle equivalent to Bitcoin's, while preserving the 168,000,000 RIN macro-cap.

## Specification

### Block subsidy function

Let `nHeight` denote the block height at which a coinbase output is produced. The block subsidy `GetBlockSubsidy(nHeight)` MUST return the value defined by the following piecewise function, expressed in satoshis (where 1 RIN = 100,000,000 satoshis):

```
GetBlockSubsidy(nHeight):
    if nHeight <       0: return 0                   # invalid
    if nHeight <  210000: return 50.00 * COIN        # Phase 0
    if nHeight <  420000: return 25.00 * COIN        # Phase 1
    if nHeight <  630000: return 12.50 * COIN        # Phase 2
    if nHeight <  840000: return  6.25 * COIN        # Phase 3
    if nHeight < 2100000: return  4.00 * COIN        # Phase 4 (CH activated)
    if nHeight < 4200000: return  2.00 * COIN        # Phase 5
    if nHeight < 6300000: return  1.00 * COIN        # Phase 6
    return 0.60 * COIN                                # Terminal phase
```

### Phase boundaries (normative)

| Phase | Block Range | Reward (RIN) | Subsidy (sat) | Duration (blocks) |
|---|---|---|---|---|
| 0 | `[0, 210000)` | 50.00 | 5,000,000,000 | 210,000 |
| 1 | `[210000, 420000)` | 25.00 | 2,500,000,000 | 210,000 |
| 2 | `[420000, 630000)` | 12.50 | 1,250,000,000 | 210,000 |
| 3 | `[630000, 840000)` | 6.25 | 625,000,000 | 210,000 |
| 4 | `[840000, 2100000)` | 4.00 | 400,000,000 | 1,260,000 |
| 5 | `[2100000, 4200000)` | 2.00 | 200,000,000 | 2,100,000 |
| 6 | `[4200000, 6300000)` | 1.00 | 100,000,000 | 2,100,000 |
| Terminal | `[6300000, ∞)` | 0.60 | 60,000,000 | perpetual |

### Cumulative supply at terminal phase entry

At the moment block 6,300,000 is mined, the cumulative supply MUST equal:

```
S(6300000) = 210000 × (50 + 25 + 12.5 + 6.25)
           + 1260000 × 4
           + 2100000 × 2
           + 2100000 × 1
           = 19,687,500 + 5,040,000 + 4,200,000 + 2,100,000
           = 31,027,500 RIN
```

This matches the value `S_fix = 31,027,500` reported in [1, §3.4].

### Activation requirement

This RIP introduces a hard fork at block 840,000 (the Phase 3→4 boundary). All consensus nodes MUST be upgraded to enforce the dilated Phase 4 subsidy of 4.00 RIN/block before block 840,000 is mined. A node enforcing the legacy Bitcoin-standard schedule would expect a 3.125 RIN subsidy at block 840,000 and reject upgraded blocks as invalid (excessive coinbase).

For deployment, this RIP specifies the schedule as **chainparams** at network genesis rather than activating via signaling, since the schedule is intended to be in effect from genesis. The `nSubsidyHalvingInterval` chainparams field and the related Customized Halving phase boundaries are initialized as specified above.

**Implementation status**: A Core v1.0.6 build by the RIP author embeds this schedule in chainparams from genesis. At the time of this RIP's most recent revision, that build's chain has progressed to block ≈ 566,944 (Phase 2, subsidy 12.50 RIN/block). Whether this constitutes adoption by the Rincoin network at large is a separate question; advancement past `Draft` requires the evidence defined in RIP-0001 §RIP Status.

### COIN constant

For the avoidance of doubt, `COIN = 100000000` (1 RIN = 10⁸ satoshis), identical to Bitcoin's convention.

## Rationale

### Why dilate at block 840,000 specifically?

Block 840,000 is the natural boundary at which Rincoin's emission schedule diverges from Bitcoin's: it is the start of what would be Phase 4 under both schedules, and it is the point at which the unmodified Bitcoin-standard schedule would yield an issuance rate (3.125 RIN/block × 525,600 blocks/yr ≈ 1.64 M RIN/yr) below the threshold at which Bitcoin itself operated during its first four years (~10 M BTC/4yr). Dilating from this point preserves all preceding phases unchanged, minimizing protocol surface area.

### Why a perpetual terminal reward of 0.6 RIN?

The terminal reward of 0.6 RIN/block contributes a fixed annual issuance of:

```
r_f = 525600 × 0.6 = 315,360 RIN/yr
```

Per [1, §3.4], this corresponds to a steady-state circulating supply of approximately:

```
C^ss = r_f / γ ≈ 315,360 / 0.0151 ≈ 20,866,000 RIN
```

at an entropic loss rate of γ ≈ 1.5%/yr. This anchors the long-run circulating supply near 21,000,000 RIN, which is the canonical reference point inherited from Bitcoin (the "Nakamoto target").

A reward of 0.5 RIN (Scenario I) yields a steady state below 21M; a reward of 1.0 RIN sustains long-run supply growth that delays convergence to the macro-cap. 0.6 RIN is the parametric value that produces near-coincidence with the Nakamoto target under the assumed loss rate.

This rationale concerns only the steady-state circulating supply contribution from the terminal subsidy. The actual long-run circulating supply also depends on the regenerative mechanisms specified in RIP-0005 (PoR), RIP-0006 (Cryptographic Vault), and RIP-0007 (Sweeper Bounty); see those RIPs for the full system equilibrium analysis.

### Alternatives considered

1. **Tail emission (Monero-style)** — Rejected. A constant non-decaying subsidy violates the fixed macro-cap of 168M RIN, conflicting with the protocol's positioning as a scarcity-preserving asset.
2. **Single dilation at block 840,000 to a large fixed reward** — Rejected. Removes the deflationary halving cadence that contributes to monetary policy predictability and store-of-value characteristics.
3. **Earlier dilation (e.g., at block 420,000)** — Rejected. Phases 0–3 are deliberately preserved at Bitcoin-standard cadence to maintain narrative continuity with the canonical halving cycle.
4. **Scenario I (0.5 RIN terminal)** vs. **Scenario II (0.6 RIN terminal)** — Scenario II is selected as baseline; see Whitepaper [1, §3.6] for comparative analysis.

### Why not specify the circuit breaker here?

The conditional fallback to a 1.0 RIN intermediate phase (Scenario III in [1, §3.5]) introduces a governance trigger and additional consensus complexity that are conceptually separable from the baseline schedule. Combining them in one RIP would conflate a deterministic specification with a conditional governance mechanism. RIP-0003 specifies the circuit breaker independently.

## Backwards Compatibility

This RIP defines a **hard fork**. A node enforcing the Bitcoin-standard 210,000-block halving interval indefinitely will, at block 840,000, expect a coinbase subsidy of 3.125 RIN and will reject any block produced under this RIP (which carries 4.00 RIN) as having an excessive coinbase output.

For Rincoin's pre-mainnet deployment scenario, this is purely theoretical: the schedule is set at genesis and no legacy nodes exist. For any deployed network, all full nodes and miners MUST upgrade before block 840,000.

## Reference Implementation

The Customized Halving schedule is implemented in **Rincoin Core**, the reference full-node implementation:

- **Official repository**: https://github.com/Rin-coin/rincoin
- **Developer fork (pre-release tags)**: https://github.com/Aevust/rincoin
- **Release**: [`core-v1.0.6`](https://github.com/Aevust/rincoin/releases/tag/core-v1.0.6) (pre-release) — first production release implementing this RIP. Operating on the official SEED servers since deployment.

The relevant logic resides in `src/validation.cpp` (function `GetBlockSubsidy`) and `src/chainparams.cpp` (`nSubsidyHalvingInterval` and the Customized Halving phase boundaries).

A regtest validation harness is provided by [`rincoin-sim`](https://github.com/Aevust/rincoin-sim), which implements the full schedule on a 1/1000 scale (i.e., `nSubsidyHalvingInterval = 210` blocks, Phase 4 activation at block 840) for accelerated regtest validation.

Mainnet deployment uses unscaled values:

- `nSubsidyHalvingInterval = 210000`
- Customized Halving activation height: `840000`
- Phase 5 boundary: `2100000`
- Phase 6 boundary: `4200000`
- Terminal phase boundary: `6300000`

These values are embedded in the genesis chainparams of Core v1.0.6 and have been enforced by every block validation since genesis.

## Security Considerations

### Hash rate transition risk

A subsidy step from 6.25 RIN to 4.00 RIN at block 840,000 represents a 36% reduction in per-block reward, larger than Bitcoin's standard 50% halving impact in absolute issuance terms (because each prior halving was already 50%, the *delta* at 840,000 is smaller proportionally than a normal halving but occurs after only ~1.6 years rather than ~4 years, when fee market maturation is unlikely). Hash rate may contract sharply.

The asymmetric step at the Phase 6 → Terminal boundary (1.00 RIN → 0.60 RIN, a 40% reduction rather than a 50% halving) introduces a non-standard discontinuity. Implementations MUST NOT round or smooth this boundary; the discrete step is intentional and any smoothing would violate the macro-cap arithmetic.

The Conditional Stability Valve (RIP-0003) is intended to mitigate the Phase 6 → Terminal transition risk if hash-rate metrics signal vulnerability prior to block 6,300,000.

### Coinbase output validation

Implementations MUST reject any block whose total coinbase output (including transaction fees) exceeds `GetBlockSubsidy(nHeight) + total_fees`. A block with a coinbase output less than the subsidy is permitted (miners may forfeit a portion of the subsidy), but the consensus rule binds only the upper bound.

### Off-by-one boundary errors

The phase boundaries in the Specification use half-open intervals `[start, end)`. Implementations MUST verify boundary blocks (e.g., block 839 vs. block 840) using strict less-than comparisons. The `rincoin-sim` test vectors above explicitly validate both sides of every boundary to detect off-by-one errors.

### Secular horizon assumption

This RIP encodes a deterministic schedule extending to block 6,300,000 (~12 years post-CH-activation, ~13.6 years post-genesis at 60 s/block) and a perpetual terminal subsidy thereafter. The mathematical analysis underlying the 0.6 RIN terminal value assumes a continuous loss rate γ ≈ 1.5%/yr persisting over a multi-century horizon. Empirical deviation of γ from this assumption shifts the steady-state equilibrium per the relation `C^ss = r_f / γ`. RIP-0005 (PoR) provides the regenerative mechanism that compensates for this drift.
